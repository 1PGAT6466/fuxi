## 一、系统概述

伏羲是一个企业知识认知体系，采用中医脏腑隐喻的"生命体"架构。系统将知识处理流程映射为人体的消化、循环、决策和表达过程，通过经络系统实现模块间的信号传递。

### 核心理念

| 概念 | 隐喻 | 技术含义 |
|------|------|----------|
| 伏羲 | 体系（System of Systems） | 由多个系统组成的完整知识认知体系 |
| 四象 | 系统（System） | 少阳、太阳、少阴、太阴四大功能系统 |
| 器官 | 组件（Component） | 系统内的具体功能模块 |
| 经络 | 信号总线 | 模块间异步通信网络 |

## 三、四象系统详解

### 3.1 少阳·消化系统（知识消化中枢）

负责文档的解析、清洗、分块和知识提取。将原始文档转化为结构化的知识片段。

```
shaoyang/
├── pipeline.py        # 统一处理管线
├── extractor.py       # SAG式事件/实体提取
├── semantic_chunker.py # 语义分块（替代原 chunker.py）
├── cleaner.py         # 文本清洗
├── parser.py          # 文档解析
└── ingest.py          # 入库管理
```

**处理流程**:

```
原始文档 → parser 解析 → cleaner 清洗 → semantic_chunker 分块 → extractor 提取 → 入库
```

### 3.2 太阳·筑基系统（精炼排序中枢）

负责检索和排序。通过多路召回、融合、精排，从海量知识中找到最相关内容。

```
taiyang/
├── retrieval.py       # 混合检索管线
├── multi_hop.py       # SAG式多跳检索
├── fusion.py          # RRF融合
├── rerank.py          # 四级降级精排
├── dynamic_alpha.py   # 动态融合权重
├── query_expansion.py # 查询扩展
├── graph.py           # 图谱检索
└── cache.py           # 语义缓存
```

**检索管线**:

```
L-1: QA对匹配 → L0: 语义缓存+图谱路由 → L1: 同义词+LLM改写 →
L1.5: HyDE → L2: BM25+向量双路 → L3: RRF融合+动态alpha →
L4: 三阶段精排 → L5: Rerank四级降级 → L6: Parent-Child+Sentence Window
```

### 3.3 少阴·炼化系统（决策合成中枢）

负责答案生成和质量控制。通过 LLM 合成答案，并进行事实性校验和质量评估。

```
shaoyin/
├── brain.py           # 决策合成引擎
├── judge_v2.py        # LLM-as-Judge评分
├── fact_check.py      # 事实性校验
├── context_compressor.py # 上下文压缩
├── strategy.py        # 策略选择
├── orchestrator.py    # 编排器
└── resolver.py        # 解析器
```

### 3.4 太阴·显化系统（对外接口中枢）

负责体系对外服务。提供 API 接口、监控指标、审计日志和安全管理。

```
taiyin/
├── server.py          # 对外接口
├── audit.py           # 审计日志
├── metrics.py         # Prometheus指标
├── security.py        # 安全模块
├── flags.py           # Feature Flag管理
├── monitor.py         # 监控模块
└── mcp_protocol.py    # MCP协议
```

## 四、经络系统

经络系统是体系内唯一的通信网络，负责系统和器官间的异步通信和状态同步。

### 信号类型

```python
class SignalType(Enum):
    HEARTBEAT = "heartbeat"  # 心跳信号
    ALERT = "alert"          # 告警信号
    DATA = "data"            # 数据信号
    COMMAND = "command"      # 命令信号
```

### 信号优先级

```python
class Priority(Enum):
    CRITICAL = 0  # 紧急
    HIGH = 1      # 高
    NORMAL = 2    # 普通
    LOW = 3       # 低
```

### 通信模式

```
器官A ──emit(signal)──▶ 经络总线 ──dispatch──▶ 器官B
器官A ◀──callback────── 经络总线 ◀──subscribe── 器官B
```

## 六、数据流

### 6.1 文档入库流程

```
用户上传 → /api/upload
  → 文件校验（类型、大小）
  → 保存到 data/uploads/
  → 少阳·消化系统处理
    → parser 解析（pdfplumber / PyPDF2 / python-docx）
    → cleaner 清洗（去除噪音、格式化）
    → chunker 分块（语义分块 / 固定分块）
    → extractor 提取（实体、事件、关键词）
  → MemoryStore 存储（SQLite FTS5）
  → ChromaDB 向量化（BGE-base-zh-v1.5）
  → 知识图谱更新
  → 审计日志记录
```

### 6.2 检索流程

```
用户查询 → /api/search?q=...
  → 大脑·brain 意图识别
  → 少阴·炼化系统 查询规划
  → 太阳·筑基系统 检索
    → L-1: QA对匹配（<10ms）
    → L0: 语义缓存+图谱路由（<50ms）
    → L1: 同义词+LLM改写
    → L1.5: HyDE 假设文档（~500ms）
    → L2: BM25+向量双路召回
    → L3: RRF融合+动态alpha
    → L4: 三阶段精排（匹配→分类→MMR去重）
    → L5: Rerank四级降级（SiliconFlow→本地→BM25→原始）
    → L6: Parent-Child+Sentence Window
  → Top-K 结果返回
```

### 6.3 对话流程

```
用户提问 → /api/chat
  → 大脑·brain 意图识别
  → 少阴·炼化系统 编排
    → 检索相关知识
    → 上下文压缩
    → LLM 生成答案（MiMo 2.5 Pro → Ollama → 规则兜底）
    → 事实性校验
    → 质量评分（LLM-as-Judge）
  → 太阴·显化系统 返回响应
  → 审计日志记录
```

## 十一、安全架构

### 认证

- JWT Token 认证（HS256 算法）
- Token 有效期：24 小时
- 白名单端点无需认证

### 输入验证

- 文件类型白名单（30+ 种格式）
- 文件大小限制（默认 200MB）
- 请求限流（60 次/分钟/IP）

### 敏感信息脱敏

系统自动识别并脱敏以下信息：
- 密码字段（password/passwd/pwd）
- Token/密钥字段（secret/token/api_key）
- 身份证号码
- 手机号码

| 条件 | 级别 | 处理 |
