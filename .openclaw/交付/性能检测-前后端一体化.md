### 3. 缓存使用情况

**🟢 LOW** — 缓存架构合理，有优化空间

**已有缓存**:
- ✅ **语义缓存 (L1+L2)**: `src/services/cache.py` 和 `src/taiyang/cache.py` — 双层缓存策略
  - L1: 精确匹配 (MD5 hash)
  - L2: 语义相似匹配 (余弦相似度 > 0.92)
  - 容量: 200 条，TTL: 1 小时
- ✅ **Redis**: 用于任务队列、会话存储、权限缓存
- ✅ **前端 GET 缓存**: `api-client.js` — 30 秒 TTL

**问题**:
- 语义缓存的 L2 层在每次 `set_cache` 时调用 `embed_texts`，增加写入延迟
- `_cosine_similarity` 用纯 Python 实现，大向量时效率低
- Redis 仅用于任务队列，未用于 RAG 查询结果缓存

**修复建议**:
- L2 缓存写入改为后台异步任务，不阻塞主流程
- 余弦相似度计算改用 `numpy` 向量化
- 考虑将热查询结果写入 Redis，跨进程共享

### 4. 连接池配置

**🔴 HIGH** — SQLite 连接池配置不当

**当前状态**:
- `connection_pool.py`: SQLite 连接池，最大 5 连接
- `http_client.py`: HTTP 连接池（aiohttp.ClientSession）
- 多个模块各自创建 HTTP session，未统一管理

**问题**:
1. **SQLite 连接池 max_connections=5 太小** — 高并发时会阻塞等待
2. **连接等待机制用 `time.sleep(0.1)` 轮询** — 在 async 上下文中阻塞事件循环
3. **HTTP session 分散管理** — `xun.py`、`http_client.py`、`llm.py` 等各自创建 session
4. **无连接池监控** — 缺少连接使用率指标

**修复建议**:
```python
# 1. 增大连接池
pool = SQLiteConnectionPool(db_path, max_connections=20)

# 2. 用 asyncio.sleep 替代 time.sleep
await asyncio.sleep(0.1)  # 而非 time.sleep(0.1)

# 3. 统一 HTTP session 管理
# 在 server.py startup 中创建全局 session，通过依赖注入分发

# 4. 添加连接池指标
# 暴露 active_connections / max_connections 到 Prometheus
```

### 5. N+1 查询检测

**🔴 HIGH** — 发现 30+ 处 N+1 查询模式

**严重案例**:

| 文件 | 行号 | 问题 |
|------|------|------|
| `agentic_rag_v2.py` | 187, 378, 383 | 循环内数据库查询 |
| `chat.py` | 480 | 循环内数据库查询（消息历史加载） |
| `data_service.py` | 130 | 循环内数据库查询 |
| `dream_cycle.py` | 388, 466, 616 | 循环内数据库查询 |
| `dui.py` | 1021, 1033 | 循环内数据库+远程调用 |
| `context_compressor.py` | 86 | 循环内数据库查询 |
| `alerting.py` | 87, 179 | 循环内远程调用+数据库查询 |
| `base_gua.py` | 313-605 | 多处循环内远程调用 |
| `auto_graph.py` | 189-557 | 循环内远程调用 |

**修复建议**:
- 数据库查询：使用 `WHERE id IN (...)` 批量查询替代循环单条查询
- 远程调用：使用 `asyncio.gather()` 并发调用替代串行循环
- 示例：
```python
# ❌ N+1
for entity_id in entity_ids:
    result = await db.execute(select(Entity).where(Entity.id == entity_id))

# ✅ 批量
results = await db.execute(select(Entity).where(Entity.id.in_(entity_ids)))
```

## 三、前后端链路

### 1. API 响应时间预估

**🔴 HIGH** — 多个链路存在性能瓶颈

**热路径分析**:

| 链路 | 预估延迟 | 瓶颈 |
|------|----------|------|
| 对话 (chat) | 2-15s | LLM 调用 + RAG 检索 + 上下文压缩 |
| 搜索 (search) | 0.5-3s | 向量检索 + BM25 + Rerank |
| 文件上传 (upload) | 1-30s | 文件大小 + 解析 + 向量化 |
| 知识图谱 (graph) | 0.3-1s | 图数据库查询 |
| Wiki 页面 | 0.2-0.5s | 数据库查询 |

**瓶颈点**:
1. **LLM 调用链过长**: `chat → orchestrator → qian → smart_router → llm` — 至少 4 层函数调用
2. **RAG 检索串行**: 向量检索 → BM25 → Rerank → 上下文压缩 — 全部串行
3. **N+1 查询**: 消息历史、实体查询等存在循环查询

**修复建议**:
- RAG 检索：向量检索和 BM25 并行执行 (`asyncio.gather`)
- LLM 调用：减少中间层，直接路由到具体 LLM provider
- 添加请求级缓存（相同 query 在短时间内的结果）

### 2. SSE 流式传输效率

**🟡 MEDIUM** — SSE 实现广泛但有优化空间

**SSE 使用范围**: 70+ 个模块使用 SSE（通过 `EventSourceResponse` 或 `text/event-stream`）

**问题**:
- `chat.py` 的 `sse_generator` 是核心流式输出，需确保 token 级别的流式
- 多个模块的 SSE 实现可能有重复代码
- 无 SSE 连接池管理（大量并发 SSE 连接可能耗尽连接数）

**修复建议**:
- 统一 SSE 中间件，减少重复代码
- 添加 SSE 连接数监控和限制
- 使用 Server-Sent Events 的 `retry` 字段控制重连间隔

### 4. 错误重试机制

**🟡 MEDIUM** — 重试机制存在但不够完善

**后端** (`retry.py`):
- ✅ `retry_async`: 指数退避，最多 3 次重试
- ✅ `retry_sync`: 同步版本
- 使用 `retry` 的模块: 19 个

**前端** (`api-client.js`):
- ✅ 超时控制: 15 秒默认超时
- ✅ 自动重试: 最多 2 次，仅 5xx 错误
- ✅ 网络错误重试

**问题**:
- 后端 `retry_async` 使用 `asyncio.sleep` 但未配置 jitter（可能导致雷群效应）
- 前端重试未区分 429 (限流) — 应尊重 `Retry-After` header
- LLM 调用重试 (`llm.py`) 有自己的重试逻辑，与 `retry.py` 不统一

**修复建议**:
- 添加 jitter: `delay * (0.5 + random.random())`
- 前端处理 429 时读取 `Retry-After` header
- 统一 LLM 调用重试逻辑到 `retry.py`
