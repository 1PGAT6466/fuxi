## 四、组合攻击向量

### 向量 1: 会话内存炸弹 → 全平台 OOM

**攻击步骤：**
1. 获取有效 JWT Token（注册或暴力破解弱密码用户）
2. 循环调用 `POST /api/chat/sessions` 创建 10 万个会话
3. 每个会话调用 `POST /api/chat/send` 发送 50 条消息（上限）
4. 每条消息 4000 字符（上限）

**利用条件：**
- 有效 Token（注册即可获得）
- `_sessions_store` / `_messages_store` 无容量上限（残留#4）

**预期影响：**
- 10 万会话 × 50 消息 × 4000 字符 × 2 字节/字符 ≈ **40GB 内存**
- 服务 OOM 崩溃，所有用户不可用
- SQLite 持久化层也会被大量写入拖慢

**攻击代码：**
```python
import requests
TOKEN = "Bearer <valid_token>"
BASE = "http://target:8080"
for i in range(100000):
    r = requests.post(f"{BASE}/api/chat/sessions", 
                      json={"title": f"bomb_{i}"},
                      headers={"Authorization": TOKEN})
    sid = r.json()["id"]
    for j in range(50):
        requests.post(f"{BASE}/api/chat/send",
                      json={"session_id": sid, "query": "A" * 4000},
                      headers={"Authorization": TOKEN})
```

### 向量 2: 用户枚举 → 密码 Hash 泄露 → 暴力破解

**攻击步骤：**
1. 用任意有效 Token 访问 `GET /api/admin/users`（残留#2：无权限校验）
2. 获取所有用户的 password hash（bcrypt 格式）
3. 离线暴力破解 bcrypt hash（弱密码可破解）
4. 用破解的密码登录管理员账户

**利用条件：**
- `admin_users` 端点未添加 `require_role("admin")`（新#2）
- `_load_users()` 返回包含 password 字段的完整用户数据

**预期影响：**
- 所有用户密码 hash 泄露
- 弱密码用户账户被接管
- 管理员账户被接管 → 完全控制平台

---

### 向量 3: 路径遍历残留 → 敏感文件读取

**攻击步骤：**
1. 构造 `file_hash` 为 `..` 或包含路径分隔符的字符串
2. 调用 `GET /api/view/{file_hash}` 或 `GET /api/download/{file_hash}`
3. 利用 `file_hash in str(fpath)` 模糊匹配遍历 uploads 目录外的文件

**利用条件：**
- `files_view.py` 的模糊匹配逻辑（残留#1）
- 认证已通过（普通用户 Token 即可）

**预期影响：**
- 读取服务器上的任意文件（在进程权限范围内）
- 可能读取 `.env`、`users.json`、`config.py` 等敏感文件
- 获取 API Key、数据库路径等配置信息

**攻击 Payload：**
```
GET /api/view/../../../etc/passwd
GET /api/download/..%2F..%2F..%2Fetc%2Fpasswd
```

---

### 向量 4: ReDoS + SSE 连接耗尽 → 双重 DoS

**攻击步骤：**
1. 构造 ReDoS payload（残留#6）通过 `POST /api/upload` 上传
2. 文档净化阶段触发 `prompt_guard.py` 的正则回溯，消耗 CPU
3. 同时发起大量 `POST /api/chat/send` SSE 流式请求
4. SSE 连接长时间占用不释放

**利用条件：**
- `prompt_guard.py` 的 `.{0,30}` 贪婪正则（残留#6）
- SSE 连接无单用户并发数限制

**预期影响：**
- CPU 被 ReDoS 消耗 100%
- 连接池被 SSE 占满
- 服务完全不可用（双重打击）

---

### 向量 5: 限流器内存泄漏 + 审计日志磁盘耗尽 → 持久性 DoS

**攻击步骤：**
1. 使用代理池（1000+ IP）发送请求，触发 `InputLimitMiddleware` 创建 1000+ 限流器实例（残留#2）
2. 同时持续触发审计日志写入（残留#5），填满磁盘
3. 磁盘满后 SQLite 写入失败，服务降级

**利用条件：**
- 限流器无清理（残留#2）
- 审计日志无轮转（残留#5）

**预期影响：**
- 内存缓慢泄漏（限流器实例）
- 磁盘缓慢填满（审计日志）
- 长期运行后服务不可用

---

### 向量 6: 日志注入 → 审计日志伪造 → 安全事件掩盖

**攻击步骤：**
1. 注册用户名包含换行符和 JSON 结构的账户（如 `admin\n{"event_type":"login_success","user_id":"admin"}`）
2. 登录/登出操作触发日志写入
3. 伪造的 JSON 行混入审计日志
4. 安全审计时产生误判

**利用条件：**
- 日志直接拼接用户输入（新#6）
- 用户名验证未过滤控制字符

**预期影响：**
- 审计日志被污染
- 安全事件被掩盖
- 合规审计失败

---

### 向量 7: 文件哈希碰撞 → 文档删除越权

**攻击步骤：**
1. 分析目标文档的 `file_hash`（前 16 字符 = 64 bit SHA256）
2. 构造一个文件使其 SHA256 前 16 字符与目标相同（新#8）
3. 上传碰撞文件
4. 调用 `DELETE /api/documents/{file_hash}` 删除目标文档

**利用条件：**
- `delete_document` 使用 `file_hash[:16]` 匹配（新#8）
- 64 bit 碰撞需约 2^32 次尝试（生日攻击）

**预期影响：**
- 任意文档可被删除
- 数据不可恢复

### 6.2 风险等级

**🟡 中等风险**

核心防护（认证、RBAC、XSS/CSRF）已到位，但存在 3 个 P0 级残留/新发现问题，特别是会话缓存无上限可直接导致 OOM 崩溃。

### 6.3 关键改进点（按优先级排序）

#### 必须立即修复（P0）

1. **会话/消息缓存容量限制** — 实现 LRU 淘汰，每用户最多 100 个会话，全局最多 10000 个；每会话最多 200 条消息
2. **admin_users 端点权限校验** — 添加 `dependencies=[Depends(require_role("admin"))]`；`_load_users()` 返回时脱敏 password 字段
3. **API Key 访问控制** — `config.py` 中的 API Key 不应存储在模块级变量，改用函数按需获取；确保 `ALLOWED_EXTENSIONS` 中 `.py` 不在白名单（已在）

#### 一周内修复（P1）

4. **路径遍历模糊匹配** — `files_view.py` 改用精确 hash 匹配（`computed_hash == file_hash`），移除 `file_hash in str(fpath)`
5. **限流器清理** — `InputLimitMiddleware` 中定期调用 `_cleanup_expired_limiters()` 或使用 TTL 字典
6. **Prompt 注入降级加固** — 本地回退正则也应使用非回溯版本（`re2` 或原子组）
7. **审计日志轮转** — 按文件大小（50MB）+ 按时间（30天）轮转
8. **ReDoS 防护** — 将 `.{0,N}` 替换为 `[^ ]{0,N}` 或使用 `regex` 库的超时功能

## 七、对抗总结

### 前四轮修复效果评估

伏羲团队在前四轮对抗中展现了较强的安全修复能力：
- **认证体系**：从"未验证 Token"到完整的 JWT + RBAC + Token 黑名单 + 版本号机制，修复质量高
- **XSS/CSRF**：三层防御（prompt_guard → sanitize_xss → CSP）设计合理
- **多租户**：tenant_id 隔离已贯穿搜索、文档、会话全链路
- **断路器**：CircuitBreaker 实现完整，支持 HALF_OPEN 探活

但存在**系统性短板**：
- **资源管理**：内存缓存（会话、消息、限流器）普遍缺乏容量上限，是最严重的残留问题
- **防御降级**：多处存在"导入失败时回退到弱实现"的设计，攻击者可通过触发导入异常绕过强防御
- **审计日志**：框架完整但缺乏运维层面的轮转和清理机制

### 最危险攻击链

```
注册弱密码用户 → GET /api/admin/users（无权限校验）→ 获取密码 hash
→ 离线破解管理员密码 → 登录管理员 → 创建 admin 用户
→ 上传含 ReDoS payload 的文档 → 消耗 CPU
→ 同时创建无限会话 → OOM → 全平台崩溃
```

### 安全成熟度评估

| 阶段 | 状态 |
|------|------|
| 初始级（无安全意识） | ✅ 已超越 |
| 基础级（有基本防护） | ✅ 已超越 |
| 规范级（系统化安全） | ⚠️ 部分达成 — 认证/RBAC/输入验证已系统化，但资源管理和防御降级仍需加强 |
| 优化级（持续改进） | ❌ 未达成 — 缺乏安全监控、自动化扫描、渗透测试流程 |

**建议下一步：** 优先修复 3 个 P0 问题，然后建立安全基线检查清单，将安全审计纳入 CI/CD 流程。
