## 一、后端架构检测

### 1. FastAPI 路由注册完整性

| 检测项 | 状态 | 详情 |
|--------|------|------|
| 自动发现机制 | ✅ 正常 | `src/api/_auto_discovery.py` 实现了文件名→前缀的自动映射 |
| 手动注册路由 | ⚠️ 冗余 | `core/routes.py` 和 `src/routes.py` 两套路由注册逻辑并存 |
| 路由冲突检测 | ⚠️ 存在 | `files_alias.py` 与 `documents.py` 存在 `/api/files` 路径重叠 |
| 服务路由完整性 | ✅ 完整 | 20+ 个路由模块全部注册，覆盖搜索/对话/文档/图谱/Wiki 等 |

**🔴 HIGH — 双路由注册模块并存**

`src/routes.py` (v1.50) 和 `src/core/routes.py` (v1.44) 两套路由注册逻辑并存，`server.py` 的 startup 事件调用 `core/routes.py`，但 `src/routes.py` 也定义了完全相同的 `register_all_routes` 函数。这会导致：
- 维护混乱：修改路由时不确定改哪个文件
- 潜在重复注册：如果两个模块都被导入，路由会重复挂载

**修复建议**: 删除 `src/routes.py`，统一使用 `src/core/routes.py` 作为唯一路由注册入口。

### 2. 依赖注入使用率

| 检测项 | 状态 | 详情 |
|--------|------|------|
| `Depends(require_admin)` | ✅ 广泛使用 | 管理端点统一使用依赖注入鉴权 |

| 数据库连接注入 | ❌ 未使用 | SQLite 连接通过 `data_service.py` 全局管理 |

**🟡 MEDIUM — 服务层缺乏依赖注入**

所有服务层（`src/services/`）通过硬编码 import 调用，没有使用 FastAPI 的依赖注入系统管理服务实例。例如：
- `chat.py` 直接 `from src.data_service import ...`
- `search.py` 直接 `from src.taiyang.retrieval import ...`
- `kb.py` 直接 `from src.db.vector_store import ...`

**影响**: 单元测试困难，无法 mock 服务依赖；服务替换需要修改调用方代码。

**修复建议**: 引入轻量级 DI 容器（如 `dependency-injector` 或自定义 `get_service()` 函数），将服务实例通过 `Depends()` 注入路由。

### 3. 中间件链配置

| 检测项 | 状态 | 详情 |
|--------|------|------|
| 安全响应头 | ✅ 完整 | CSP、HSTS、X-Frame-Options 等全部配置 |
| CORS | ✅ 正确 | 允许方法和头合理配置 |
| GZip 压缩 | ✅ 已启用 | 最小 500 字节触发 |
| 认证中间件 | ✅ 完整 | JWT 验证 + 白名单 + 输入限流 |
| 引擎路由中间件 | ✅ 已实现 | v1/v2 引擎自动检测 |
| 请求指标中间件 | ✅ 已实现 | 延迟 + 错误率统计 |
| 限流中间件 | ✅ 双层 | slowapi 全局限流 + InputLimitMiddleware 滑动窗口 |

**🔴 HIGH — 中间件双重定义**

`server.py` 和 `src/middleware.py` 都定义了完全相同的中间件链（安全头、认证、CORS、GZip、引擎路由、指标、限流）。`server.py` 的 startup 事件调用 `register_all_routes(app)`，但中间件在 `server.py` 模块级别直接注册。如果 `middleware.py` 的 `setup_middleware(app)` 也被调用，会导致：
- 中间件重复执行（每个请求经过两次安全头注入、两次认证检查）
- 性能下降
- 潜在的响应头冲突

**修复建议**: 删除 `src/middleware.py`，统一在 `server.py` 中注册中间件；或将中间件注册逻辑迁移到 `middleware.py` 并从 `server.py` 调用 `setup_middleware(app)`。

**🟡 MEDIUM — metrics_middleware 中存在 dead code**

`src/middleware.py` 第 79 行有连续两个 `raise`：
```python
            raise
            raise  # dead code
```
第二个 `raise` 永远不会执行。

| 未捕获异常处理 | ⚠️ 缺失 | 无通用 `Exception` 处理器 |

**🟡 MEDIUM — 缺少通用 Exception 处理器**

当前异常处理器覆盖了 HTTPException、ValidationError、RateLimitExceeded，但没有通用的 `Exception` 处理器。未预期的异常（如 `RuntimeError`、`ValueError`）会返回 FastAPI 默认的 500 响应，格式不统一。

**修复建议**: 添加通用异常处理器：
```python
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    logger.exception(f"未处理异常: {exc}")
    return JSONResponse(
        status_code=500,
        content={"status": "error", "message": "服务器内部错误", "status_code": 500}
    )
```

```typescript
});
```

## 三、前后端耦合检测

### 1. API 契约一致性

| 检测项 | 状态 | 详情 |
|--------|------|------|
| 响应格式统一 | ⚠️ 双模式 | 后端支持旧格式 + v2 格式（`?format=v2`） |
| 前端期望格式 | ⚠️ 不一致 | 前端 `types/index.ts` 定义 `ApiResponse<T>` 含 `code` 字段，后端无 `code` 字段 |
| 登录响应 | ⚠️ 不匹配 | 前端期望 `{token, user: UserInfo}`，后端返回 `{token, username, role, display_name}` |
| 搜索响应 | ⚠️ 不匹配 | 前端期望 `SearchResponse {results, total}`，后端返回 `{wiki_results, chunk_results, results, query, page, page_size, total}` |
| 文件列表 | ⚠️ 不匹配 | 前端期望 `FileListResponse {files, total}`，后端返回 `{files, total, page, page_size}` |
| Chat 会话 | ✅ 一致 | 前端 `ChatSession` 与后端 `/api/chat/sessions` 返回格式匹配 |

**🔴 HIGH — API 响应格式双模式导致前端解析不一致**

后端同时维护两套响应格式：
- **旧格式**（默认）：直接返回业务数据 `{files: [...], total: N}`
- **v2 格式**（需显式请求）：统一包装 `{status: "success", message: "ok", data: {...}}`

前端 `api/index.ts` 的响应拦截器直接返回 `response.data`（axios 解包一层），但后端旧格式返回的顶层字段（如 `files`、`results`）直接可用，而 v2 格式需要再取 `.data`。

**问题**:
1. 前端 `types/index.ts` 定义 `ApiResponse<T>` 含 `code` 字段，但后端从未返回 `code` 字段
2. 前端 `auth.ts` 的 `login()` 直接访问 `data.token`，依赖旧格式
3. 前端 `files.ts` 的 `fetchFiles()` 访问 `data.files`，依赖旧格式

**修复建议**: 
1. 统一切换到 v2 格式，前端 api-client 统一解包 `.data`
2. 或删除 v2 格式支持，保持旧格式一致

### 2. 数据格式匹配

| 检测项 | 状态 | 详情 |
|--------|------|------|
| 日期格式 | ⚠️ 不一致 | 后端返回 Unix 时间戳，前端部分字段期望 ISO 字符串 |
| 文件大小 | ⚠️ 缺失 | 后端 `/api/files` 不返回 `size` 字段，前端 `files.ts` 硬编码 `size: 0` |
| 上传时间 | ⚠️ 缺失 | 后端不返回 `uploadedAt`，前端 `files.ts` 用 `new Date().toISOString()` 替代 |
| 用户信息 | ⚠️ 部分匹配 | 后端返回 `{username, role}`，前端期望 `{id, username, display_name, role, avatar, email}` |
| 分页格式 | ✅ 一致 | 后端返回 `{page, page_size, total}`，前端正确使用 |

**🟡 MEDIUM — 文件元数据字段缺失**

后端 `/api/files` 返回的文件信息缺少 `size`、`uploadedAt`、`type` 等字段，前端被迫使用占位值。这会导致：
- 文件列表显示"0B"大小
- 上传时间显示为当前时间（而非实际上传时间）

**修复建议**: 后端在文件入库时记录元数据（大小、时间、MIME类型），并在 `/api/files` 响应中返回。

```python
class ErrorCode(str, Enum):

    FILE_TOO_LARGE = "FILE_TOO_LARGE"

```
并在响应中返回 `{status: "error", code: "AUTH_EXPIRED", message: "..."}`
