**检测时间**：2026-07-10

## 1. 深层安全对抗

### 1.1 JWT Token 伪造尝试

#### 1.1.1 攻击向量：HS256 算法切换为 none

**攻击描述**：尝试将 JWT header 中的 alg 从 HS256 改为 none，绕过签名验证。

**代码位置**：`src/api/auth.py:verify_jwt_token()`

```python
payload = jwt.decode(token, _JWT_SECRET, algorithms=[JWT_ALGORITHM])
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：`algorithms=[JWT_ALGORITHM]` 明确限制了只接受 HS256，PyJWT 库会拒绝 none 算法。

---

#### 1.1.2 攻击向量：弱密钥暴力破解

**攻击描述**：当 FUXI_JWT_SECRET 未设置时，系统自动生成随机密钥。但如果环境变量被设置为弱值，攻击者可尝试暴力破解。

**代码位置**：`src/config.py`

```python
_WEAK_SECRETS = {
    "fuxi-v1.50-jwt-production-key-change-in-prod",
    "fuxi-v1.44-jwt-secret",
    "change-me", "secret", "jwt-secret", "your-secret-key", "super-secret",
}
if JWT_SECRET in _WEAK_SECRETS:
    _JWT_SECRET_FROM_ENV = _secrets.token_hex(32)
    JWT_SECRET = _JWT_SECRET_FROM_ENV
```

**对抗结果**：⚠️ **部分成功**（存在绕过可能）

**分析**：
- ✅ 已知弱密钥黑名单有效
- ❌ 黑名单不完整 — 缺少常见弱密钥如 `password`、`123456`、`admin`、`test` 等
- ❌ 未检查密钥熵 — 一个 16 字符的随机字符串仍可能被暴力破解
- ❌ 生产环境未强制要求环境变量 — 如果 .env 文件被泄露，密钥随之暴露

**修复建议**：
1. 扩展弱密钥黑名单，添加 Top 100 常见弱密钥
2. 增加密钥最小熵检查（至少 256 bits）
3. 生产环境强制要求 FUXI_JWT_SECRET 环境变量，未设置时拒绝启动

---

#### 1.1.3 攻击向量：Token 版本号绕过

**攻击描述**：尝试构造一个 token_version 超大的 Token，使版本号检查失效。

**代码位置**：`src/api/auth.py:verify_jwt_token()`

```python
token_tv = payload.get("tv")
if username and token_tv is not None:
    current_tv = get_token_version(username)
    if token_tv < current_tv:
        raise HTTPException(401, "Token 已失效，请重新登录")
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- 版本号检查是 `<` 而非 `!=`，这意味着只有递增才能使旧 Token 失效
- 攻击者无法预测未来的版本号
- 即使构造超大版本号，也只能通过当前检查，无法绕过

---

#### 1.1.4 攻击向量：Token 黑名单内存存储缺陷

**攻击描述**：利用 Token 黑名单的内存存储特性，在服务重启后旧 Token 重新生效。

**代码位置**：`src/api/auth.py`

```python
_token_blacklist: dict = {}  # token_jti → expiry_timestamp
```

**对抗结果**：✅ **成功**（漏洞确认）

**分析**：
- Token 黑名单存储在内存中，服务重启后清空
- 攻击者可在服务重启后使用已登出的 Token
- 虽然 token_versions 也是内存存储，但重启后版本号归零，旧 Token 的 tv=0 会等于 current_tv=0，通过检查

**严重程度**：🔴 **高危**

**修复建议**：
1. 将 Token 黑名单持久化到 Redis 或 SQLite
2. 或者在 JWT_SECRET 中加入服务实例 ID，重启后自动使所有旧 Token 失效

---

### 1.2 多租户边界突破

#### 1.2.1 攻击向量：tenant_id 注入

**攻击描述**：在注册时指定任意 tenant_id，绕过租户隔离。

**代码位置**：`src/api/auth_routes.py:register()`

```python
tenant_id = body.tenant_id if hasattr(body, 'tenant_id') and body.tenant_id else "default"

users[body.username] = {
    "password": _hash_password(body.password),
    "role": "user",
    "tenant_id": tenant_id,
    ...
}
```

**对抗结果**：✅ **成功**（漏洞确认）

**分析**：
- 注册端点允许用户自行指定 tenant_id
- 没有验证 tenant_id 是否存在于 tenants.json 中
- 攻击者可注册到任意租户，访问该租户的所有数据

**严重程度**：🔴 **高危**

**修复建议**：
1. 注册时验证 tenant_id 是否存在于 TenantManager 中
2. 默认只允许注册到 "default" 租户
3. 租户分配应由管理员操作

---

#### 1.2.2 攻击向量：默认租户数据泄露

**攻击描述**：非默认租户通过搜索接口访问默认租户的数据。

**代码位置**：`src/api/search.py:_filter_by_tenant()`

```python
def _filter_by_tenant(results: list, tenant_id: str) -> list:
    if tenant_id == "default":
        return results  # 默认租户可以看到所有数据
    filtered = []
    for r in results:
        meta = r.get("metadata", {})
        r_tenant = meta.get("tenant_id", "default")
        if r_tenant == tenant_id:
            filtered.append(r)
    return filtered
```

**对抗结果**：⚠️ **部分成功**（设计缺陷）

**分析**：
- 默认租户可以访问所有无 tenant_id 标记的数据
- 但非默认租户无法访问默认租户的数据（这是正确的）
- 问题在于：如果数据没有 tenant_id 标记，默认租户可以看到所有数据

**修复建议**：
1. 所有数据必须有 tenant_id 标记
2. 默认租户只能访问 tenant_id="default" 的数据

---

### 1.3 权限提升攻击

#### 1.3.1 攻击向量：JWT role 字段篡改

**攻击描述**：修改 JWT 中的 role 字段从 user 提升为 admin。

**代码位置**：`src/api/auth.py:AuthMiddleware`

```python
request.state.role = payload.get("role", "user")
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- JWT 使用 HS256 签名，攻击者无法在不知道密钥的情况下修改 payload
- 任何修改都会导致签名验证失败

---

#### 1.3.2 攻击向量：RBAC 角色列表注入

**攻击描述**：在 JWT 的 roles 字段中注入 admin 角色。

**代码位置**：`src/api/auth.py:create_jwt_token()`

```python
rbac = get_rbac()
roles = rbac.get_roles_for_token(username)
payload = {
    "sub": username,
    "role": role,
    "roles": roles,  # 来自 RBAC 系统
    ...
}
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- roles 字段在签发时从 RBAC 系统获取，不是用户可控的
- 即使攻击者修改了 roles 字段，签名验证也会失败

## 2. 深层性能对抗

### 2.1 并发竞态条件

#### 2.1.1 攻击向量：Token 刷新竞态

**攻击描述**：多个并发请求同时触发 Token 刷新，导致多个新 Token 被签发。

**代码位置**：`frontend/js/api-client.js`

```javascript
var __refreshLock = null;  // Promise | null

async function _refreshToken() {
    if (__refreshLock) {
        try {
            return await __refreshLock;
        } catch(e) {
            throw e;
        }
    }
    _acquireRefreshLock();
    ...
}
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- 前端实现了互斥锁，确保只有一个刷新请求在进行
- 其他请求等待同一个 Promise 的结果

---

#### 2.1.2 攻击向量：RBAC 单例竞态

**攻击描述**：并发请求同时触发 RBAC 实例初始化，导致多个实例被创建。

**代码位置**：`src/auth/rbac.py`

```python
_rbac_instance: Optional[RBAC] = None

def get_rbac() -> RBAC:
    global _rbac_instance
    if _rbac_instance is None:
        _rbac_instance = RBAC()
    return _rbac_instance
```

**对抗结果**：⚠️ **部分成功**（竞态窗口存在）

**分析**：
- 没有线程锁保护单例初始化
- 在极端并发情况下，可能创建多个 RBAC 实例
- 但由于 Python GIL，实际竞态概率很低

**修复建议**：
1. 添加 threading.Lock 保护单例初始化
2. 或使用 `__new__` 方法实现线程安全的单例

---

### 2.2 内存泄漏构造

#### 2.2.1 攻击向量：速率限制器内存泄漏

**攻击描述**：大量不同 IP 的请求创建大量 SlidingWindowRateLimiter 实例，耗尽内存。

**代码位置**：`src/api/auth.py:InputLimitMiddleware`

```python
def _get_limiter(self, key: str, max_req: int, window: int):
    if key not in self._limiters:
        with self._lock:
            if key not in self._limiters:
                from src.infra.rate_limiter import SlidingWindowRateLimiter
                self._limiters[key] = SlidingWindowRateLimiter(max_req, window)
    return self._limiters[key]
```

**对抗结果**：⚠️ **部分成功**（存在风险）

**分析**：
- 每个不同的 IP+端点组合会创建一个新的限流器实例
- 没有清理机制，长期运行可能导致内存增长
- 但实际影响有限，因为 IP 数量通常是有限的

**修复建议**：
1. 添加 TTL 清理机制，定期清理过期的限流器实例
2. 限制最大实例数量

---

#### 2.2.2 攻击向量：Token 黑名单内存泄漏

**攻击描述**：大量 Token 被加入黑名单，但清理机制不够积极，导致内存增长。

**代码位置**：`src/api/auth.py`

```python
def _cleanup_blacklist() -> None:
    now = time.time()
    with _blacklist_lock:
        expired = [k for k, v in _token_blacklist.items() if now > v]
        for k in expired:
            del _token_blacklist[k]
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- 清理机制在每次添加黑名单时触发
- 过期条目会被自动清理
- 但如果大量 Token 同时过期，清理可能不够及时

---

## 3. 深层前端对抗

### 3.1 XSS 绕过净化

#### 3.1.1 攻击向量：DOMPurify 绕过

**攻击描述**：尝试绕过前端的 HTML 转义函数 `esc()`。

**代码位置**：`frontend/js/utils.js`

```javascript
function esc(s) {
    if (!s) return '';
    var div = document.createElement('div');
    div.appendChild(document.createTextNode(String(s)));
    return div.innerHTML;
}
```

**对抗结果**：❌ **失败**（防御有效）

**分析**：
- 使用 DOM API 进行转义，这是最安全的方法
- 无法通过任何输入绕过

#### 3.1.2 攻击向量：Markdown 渲染 XSS

**攻击描述**：在 Markdown 内容中注入恶意脚本。

**代码位置**：`frontend/js/chat.js`

```javascript
appendMsg('ai', answer, d.sources, d.trace);
```

**对抗结果**：⚠️ **部分成功**（存在风险）

### 3.2 CSRF 攻击

#### 3.2.1 攻击向量：CSRF Token 绕过

**攻击描述**：尝试绕过 CSRF Token 验证。

**代码位置**：`frontend/js/api-client.js`

```javascript
opt.headers['X-CSRF-Token'] = getCSRFToken();
```

**对抗结果**：⚠️ **部分成功**（存在设计缺陷）

**分析**：
- CSRF Token 存储在 sessionStorage 中
- 但后端没有验证 CSRF Token！前端只是发送了 Token，后端没有检查
- 这意味着 CSRF 防护实际上无效

**严重程度**：🟠 **中危**

**修复建议**：
1. 后端中间件验证 X-CSRF-Token header
2. 或使用 SameSite Cookie 属性

### 3.3 前端状态篡改

#### 3.3.1 攻击向量：sessionStorage 篡改

**攻击描述**：通过浏览器控制台修改 sessionStorage 中的用户信息。

**代码位置**：`frontend/js/api-client.js`

```javascript
function getUser() {
    try { return JSON.parse(__STORE.getItem(__UK) || '{}'); } catch(e) { return {}; }
}
```

**对抗结果**：⚠️ **部分成功**（前端可篡改，但后端验证）

**分析**：
- 用户可以在浏览器控制台修改 sessionStorage 中的 user 对象
- 但前端路由守卫只检查前端存储的角色
- 后端 API 有独立的 JWT 验证，不会被前端篡改影响

**修复建议**：
1. 前端路由守卫应该验证 JWT 中的角色，而不是 sessionStorage 中的 user 对象

## 4. 集成对抗

### 4.1 多漏洞组合攻击

#### 4.1.1 攻击链：租户注入 + 数据泄露

**攻击描述**：
1. 注册时指定目标租户的 tenant_id
2. 使用新账号登录，获取该租户的 JWT
3. 调用搜索 API，访问目标租户的数据

**攻击步骤**：
```
POST /api/auth/register
{
    "username": "attacker",
    "password": "Attacker123",
    "tenant_id": "target_tenant"
}

POST /api/auth/login
{
    "username": "attacker",
    "password": "Attacker123"
}

GET /api/search?q=敏感信息
Authorization: Bearer <token_with_target_tenant>
```

**对抗结果**：✅ **成功**（攻击链完整）

**分析**：
- 步骤 1 成功：注册端点允许指定任意 tenant_id
- 步骤 2 成功：登录返回包含目标租户的 JWT
- 步骤 3 成功：搜索 API 使用 JWT 中的 tenant_id 过滤结果

**严重程度**：🔴 **严重**

**修复建议**：
1. 注册时验证 tenant_id 是否存在
2. 默认只允许注册到 "default" 租户

---

#### 4.1.2 攻击链：服务重启 + Token 复活

**攻击描述**：
1. 攻击者获取管理员的 Token
2. 管理员登出（Token 被加入黑名单）
3. 等待服务重启
4. 使用旧 Token 访问系统

**攻击步骤**：
```
# 1. 获取管理员 Token（通过其他方式）
TOKEN="eyJ..."

# 2. 管理员登出
POST /api/auth/logout
Authorization: Bearer $TOKEN

# 3. 等待服务重启

# 4. 使用旧 Token
GET /api/admin/stats
Authorization: Bearer $TOKEN
```

**对抗结果**：✅ **成功**（攻击链完整）

**分析**：
- Token 黑名单存储在内存中，重启后清空
- token_versions 也存储在内存中，重启后归零
- 旧 Token 的 tv=0 等于 current_tv=0，通过检查

**严重程度**：🔴 **严重**

**修复建议**：
1. 持久化 Token 黑名单
2. 或在 JWT_SECRET 中加入服务实例 ID

---

### 4.2 数据泄露链路

#### 4.2.1 攻击向量：错误信息泄露

**攻击描述**：通过错误响应获取系统内部信息。

**代码位置**：`src/api/auth_routes.py:register()`

```python
except Exception as e:
    logger.exception(f"register 失败: {e}")
    return server_error("注册服务异常", detail=str(e))
```

**对抗结果**：⚠️ **部分成功**（开发环境泄露）

**分析**：
- 开发环境会返回详细的错误信息（包括堆栈跟踪）
- 生产环境通过全局异常处理器脱敏
- 但某些端点（如 register）直接返回 `detail=str(e)`，可能泄露内部路径

**修复建议**：
1. 所有端点统一使用 `server_error()` 函数，不传递 detail
2. 或在生产环境中过滤 detail 字段

## 5. 总结

### 5.1 漏洞统计

| 严重程度 | 数量 | 描述 |
|---------|------|------|
| 🔴 严重 | 2 | 租户注入、Token 复活 |
| 🟠 中危 | 2 | CSRF 无效、错误信息泄露 |
| 🟡 低危 | 3 | 弱密钥黑名单不完整、RBAC 单例竞态、Markdown XSS |

### 5.2 优先修复建议

#### P0（立即修复）
1. **租户注入漏洞**：注册时验证 tenant_id 是否存在

#### P1（本周修复）
3. **CSRF Token 后端验证**：中间件验证 X-CSRF-Token
4. **错误信息脱敏**：统一错误处理，不泄露内部信息
5. **弱密钥黑名单扩展**：添加更多常见弱密钥

#### P2（本月修复）
6. **RBAC 单例线程安全**：添加 threading.Lock
7. **Markdown XSS 防护**：配置 marked sanitize 或二次净化
8. **前端路由守卫加固**：验证 JWT 而非 sessionStorage

### 5.3 安全架构建议

1. **零信任架构**：所有请求必须经过认证和授权
2. **最小权限原则**：默认拒绝，显式授权
3. **纵深防御**：多层安全检查，不依赖单一机制
4. **安全审计**：记录所有敏感操作，支持事后追溯
5. **定期渗透测试**：每季度进行一次红队检测

**检测结论**：伏羲 v1.44 整体安全水平良好，核心认证机制（JWT 签名、RBAC）设计合理。主要风险集中在多租户隔离和 Token 生命周期管理两个方面。建议优先修复 P0 级漏洞，防止数据泄露和权限绕过。
