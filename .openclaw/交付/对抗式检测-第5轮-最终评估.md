3. 将 `.env` 加入 `.gitignore`

#### 🔴 S-003: `audit.py` 数据库连接未关闭（连接泄漏）

**文件**: `src/taiyin/audit.py` 第61-75行  
**问题**: `log_audit()` 函数中 `conn = _get_conn()` 后虽有 `conn.close()`，但无 `try/finally` 保护。如果 `conn.execute()` 抛异常，连接将泄漏。

```python
def log_audit(...):
    try:
        conn = _get_conn()
        conn.execute(...)  # 如果这里抛异常
        conn.commit()
        conn.close()       # 这行不会执行
    except Exception:
        logger.warning(...)
```

**对抗尝试**: ✅ 成功 — 通过代码审查发现  
**修复建议**: 使用 `with _get_conn() as conn:` 或 `try/finally`

#### 🟠 S-005: Token 刷新端点竞态条件

**文件**: `src/api/auth_routes.py` `auth_refresh()` 函数  
**问题**: 刷新 Token 时先验证旧 Token → 加入黑名单 → 签发新 Token。在并发场景下，同一个旧 Token 可能被多次刷新，产生多个新 Token。

```python
# 竞态窗口：旧 Token 在验证通过后、加入黑名单前，可以被再次使用
payload = verify_jwt_token(token)  # ← 通过
# ... 其他线程也可以在这里通过验证
_blacklist_token(old_jti, old_exp)  # ← 还没执行
```

**对抗尝试**: ⚠️ 部分成功 — 理论上可行，但需要高并发条件  
**修复建议**: 使用分布式锁或原子操作

---

#### 🟠 S-006: 文件上传路径穿越防护可被符号链接绕过

**文件**: `src/api/documents.py` `upload()` 函数  
**问题**: 防护逻辑使用 `os.path.basename()` + `resolve()` 检查路径，但如果攻击者在上传目录中预先创建符号链接，`resolve()` 会跟随符号链接解析到目标路径，导致路径检查通过。

```python
safe_name = os.path.basename(file.filename)
tmp_path = (tmp_dir / safe_name).resolve()  # 跟随符号链接
allowed_base = tmp_dir.resolve()
if not str(tmp_path).startswith(str(allowed_base)):  # 可能被绕过
```

**对抗尝试**: ⚠️ 部分成功 — 需要预先在服务器上创建符号链接  
**修复建议**: 使用 `os.open()` + `O_NOFOLLOW` 或检查 `os.path.islink()`

#### 🟡 S-010: `files_view.py` 文件遍历性能问题

**文件**: `src/api/files_view.py` `view_document()` 函数  
**问题**: 通过 `os.walk()` 遍历整个上传目录来查找文件，对每个文件计算 SHA256 哈希。当文件数量大时，性能极差。

```python
for root, dirs, files in os.walk(UPLOAD_DIR):
    for fname in files:
        content = await asyncio.to_thread(lambda: open(fpath, "rb").read())  # 全量读取
        computed_hash = hashlib.sha256(content).hexdigest()[:16]  # 计算哈希
```

**对抗尝试**: ✅ 成功 — 通过代码审查发现  
**修复建议**: 建立文件索引（SQLite），避免全量遍历

---

#### 🟡 S-011: `antenna/search` 使用同步 `urllib.request` 发起外部请求

**文件**: `src/api/files_view.py` `antenna_search()` 函数  
**问题**: Brave Search API 调用使用同步 `urllib.request.urlopen()`，通过 `asyncio.to_thread()` 包装。虽然不会阻塞事件循环，但：
1. 没有连接池复用
2. 没有超时重试机制
3. Brave API Key 通过 URL 参数传递（可能被日志记录）

**对抗尝试**: ✅ 成功  
**修复建议**: 使用 `src.core.http_client` 的异步 HTTP 客户端

## 四、架构验证

### 4.1 中间件统一性

**结果**: ✅ 基本通过

- `middleware.py` 统一管理所有中间件（安全头、认证、CORS、GZip、引擎路由、指标、限流）
- 中间件执行顺序合理：安全头 → 认证 → CORS → GZip → 引擎路由 → 指标 → 限流
- 存在两套限流系统（`InputLimitMiddleware` + `slowapi`），可能冲突

**建议**: 移除 `slowapi` 依赖，统一使用 `InputLimitMiddleware`

### 4.2 路由去重验证

**结果**: ⚠️ 部分通过

- `_auto_discovery.py` 实现了自动路由发现
- 但 `routes.py` 中仍有大量手动注册的路由
- 存在路由前缀重复注册的风险（如 `/api/admin/status` 和 `/api/admin/server-status`）

**建议**: 完全迁移到自动发现，移除手动注册

### 4.3 API 响应格式统一性

**结果**: ⚠️ 部分通过

- `response.py` 定义了统一格式 `{status, message, data}`
- 但大量端点仍然返回旧格式（如 `{"ok": True, "files": [...]}`)
- 通过 `?format=v2` 或 `X-API-Format: v2` 头支持新格式
- 向后兼容设计合理，但增加了维护复杂度

**建议**: 设定迁移时间表，逐步统一切换到新格式

**结果**: ✅ 通过

- `frontend/dist/` 目录存在时使用构建产物
- 不存在时回退到原始 `frontend/` 目录（安全模式）

## 九、最终结论

伏羲 v1.44 的安全防护体系已经**基本成型**，核心认证、授权、输入验证、Prompt 注入防御等关键环节都有相应措施。但在**数据持久化**（黑名单/限流器内存存储）、**敏感信息管理**（.env 硬编码密码）、**连接管理**（审计日志连接泄漏）等方面仍存在需要修复的问题。

**部署建议**:
1. 修复 P0 项后可进行内网测试部署
2. 修复 P1 项后可进行有限用户试用
3. 修复 P2 项后可进行正式生产部署

**风险声明**: 本检测基于静态代码审计，未进行实际运行时测试。部分漏洞（如 S-005 竞态条件、S-006 符号链接绕过）需要特定环境条件才能触发。
