#### F-1: `src/core/http_client.py` 缺失但被引用

**位置**: `src/core/routes.py` L179-185, L192  
**现象**: `from src.core.http_client import get_session` 在 `proxy_loader_files` 和 `proxy_loader_upload` 中被导入，但 `src/core/http_client.py` 文件不存在（读取返回 ENOENT）。

#### F-2: 双重登录页导致体验割裂

**位置**: `frontend/index.html` 内嵌登录 vs `frontend/login.html` 独立登录  
**现象**: `index.html` 包含完整的登录/注册流程（JWT + token 加密存储），`login.html` 是独立的轻量登录页（无 token 加密）。路由 `/login` 指向 `login.html`，`/` 指向 `index.html`。两套登录逻辑独立实现，token 存储方式不一致。  
**影响**: 用户可能通过不同入口登录，token 管理混乱；`login.html` 缺少 AES-GCM 加密存储。  
**修复**: 统一为单入口，删除 `login.html` 或将其重定向到 `index.html` 的登录模块。

## 二、性能

### 🔴 HIGH

#### P-1: 向量查询并发信号量过低

**位置**: `src/services/retrieval.py` L18  
**现象**: `_VECTOR_SEM = asyncio.Semaphore(2)` — 仅允许 2 个并发向量查询。在多用户并发搜索时，第 3 个请求将阻塞等待。  
**影响**: 5+ 用户同时搜索时延迟显著增加（P95 可能超 10s）。  
**建议**: 提升至 `Semaphore(5)` 或根据 ChromaDB 实例性能动态调整。同时配合 `src/infra/concurrency.py` 的 `ConcurrencyManager` 统一管理。

### 🟡 MEDIUM

#### P-2: SQLite 连接管理碎片化

**位置**:  
- `src/core/db.py` → `connect()` context manager
- `src/infra/connection_pool.py` → `SQLiteConnectionPool` 全局单例
- `src/db/memory_store.py` → `_db_conn` 实例级长连接
- `src/data_service.py` → `_connect()` 每次新建连接

**现象**: 4 种不同的 SQLite 连接管理方式并存，无统一池化。`memory_store.py` 使用 `check_same_thread=False` 的长连接，其他使用短连接。  
**影响**: 连接数不可控，高并发时可能触发 `database is locked`。  
**建议**: 统一使用 `connection_pool.py` 的 `SQLiteConnectionPool`，所有模块通过 `get_connection_pool()` 获取连接。

#### P-3: LLM 调用缓存仅内存级

**位置**: `src/services/llm.py` → `_ai_cache`  
**现象**: LLM 响应缓存使用 `asyncio.Lock` 保护的 dict，重启即丢失。无缓存大小限制（仅有 lock 保护）。  
**建议**: 添加 LRU 上限（如 500 条），或迁移到 Redis 缓存层。

#### P-4: 语义缓存 L2 遍历效率

**位置**: `src/services/cache.py` → `get_cache` L2 分支  
**现象**: L2 语义匹配遍历整个 `_l2_cache` 列表（最大 200 条），每条需计算余弦相似度。当缓存接近满时，单次查询需 200 次向量运算。  
**建议**: 使用 FAISS 或 Annoy 索引加速 L2 匹配，或降低 `MAX_CACHE_SIZE` 到 100。

#### P-5: 前端未构建，运行时加载原始文件

**位置**: `frontend/` 目录  
**现象**: `vite.config.js` 存在但无 `dist/` 目录。`_register_static_routes` 中检查 `dist/` 不存在后，直接挂载原始 `frontend/` 目录（含 `.js`、`.json`、`.md` 源文件）。  
**影响**:  
1. 每个 JS 文件独立加载（无 bundle），HTTP 请求数多
2. `.md`、`.json`、`package.json` 等敏感文件可通过 `/static/` 访问（虽有 `_SafeStaticFiles` 过滤，但 `.js` 未过滤）
3. 无代码压缩，CSS/JS 体积未优化

**建议**: 执行 `npm run build` 生成 `dist/`，或部署时使用 Vue3 迁移版。

### 🟢 LOW

#### P-6: `MemoryStore` LRU 缓存线程安全

**位置**: `src/db/memory_store.py` → `_cache_hash`, `_cache_name`  
**现象**: 使用 `threading.Lock` 保护 `OrderedDict`，但 FastAPI 是异步框架，`threading.Lock` 在 `asyncio` 事件循环中可能造成阻塞。  
**建议**: 使用 `asyncio.Lock` 或确保所有缓存操作通过 `run_in_executor` 在线程池执行。

#### P-7: 文件上传无进度反馈

**位置**: `frontend/js/files.js` → 文件上传  
**现象**: 上传大文件（最大 200MB）时无进度条，用户无法判断上传状态。  
**建议**: 使用 `XMLHttpRequest.upload.onprogress` 或 `fetch` + `ReadableStream` 实现进度反馈。

## 三、架构

### 🔴 HIGH

#### A-1: `FAKE-ASYNC` 模式滥用

**位置**: 全项目约 40+ 处  
**现象**: 大量函数标记为 `async` 但内部同步执行（通过 `asyncio.to_thread` 或直接调用）。代码注释标记为 `# FAKE-ASYNC`。  
**示例**:
```python
@router.get("/api/documents")
async def documents(...):
    chunks = await asyncio.to_thread(load_chunks)  # 正确
    # 但 load_chunks() 内部是同步的
```

**影响**:  
1. 每个请求占用事件循环线程直到 `to_thread` 完成
2. `asyncio.to_thread` 使用默认线程池（ThreadPoolExecutor），默认线程数 = min(32, os.cpu_count() + 4)
3. 高并发时线程池耗尽，所有请求阻塞

**建议**:  
- 对 I/O 密集操作（SQLite 查询）统一使用 `run_in_executor` + 自定义线程池
- 或迁移到 `aiosqlite` 实现真正的异步 SQLite
- 配置自定义 `ThreadPoolExecutor(max_workers=20)` 替代默认池

### 🟡 MEDIUM

#### A-2: 错误处理过于宽泛

**位置**: 全项目约 80+ 处 `except Exception`  
**现象**: 大量 `except Exception as e: # TODO: Narrow exception type` 捕获所有异常，隐藏了真正的错误类型。  
**影响**:  
1. 调试困难：无法区分网络错误、权限错误、数据格式错误
2. 安全风险：可能捕获 `SystemExit`、`KeyboardInterrupt` 等不应被捕获的异常

**建议**: 按模块分类收窄异常类型：
- 数据库操作：`except (sqlite3.Error, OSError)`
- 网络请求：`except (aiohttp.ClientError, asyncio.TimeoutError)`
- 文件操作：`except (OSError, IOError)`

#### A-3: 中间件链顺序依赖

**位置**: `src/middleware.py`  
**现象**: 中间件注册顺序为：安全头 → Auth → InputLimit → CORS → GZip → 引擎路由 → 请求指标 → 限流。  
**问题**:  
1. `AuthMiddleware` 在 `CORSMiddleware` 之前，预检请求（OPTIONS）会被认证拦截
2. `security_headers_middleware` 使用 `@app.middleware("http")`（函数式），而 `AuthMiddleware` 使用 `add_middleware`（类式），执行顺序不直观

**建议**: 将 CORS 中间件移到所有认证中间件之前，确保 OPTIONS 预检请求不被拦截。

#### A-4: JWT 密钥双重生成

**位置**:  
- `src/config.py` L49-59 → `_JWT_SECRET_FROM_ENV` + `secrets.token_hex(32)`
- `src/api/auth.py` L57-65 → `_JWT_SECRET` + `secrets.token_hex(32)`

**现象**: 两个模块各自独立检查 `FUXI_JWT_SECRET` 环境变量并生成随机密钥。如果环境变量未设置，两处生成的密钥不同，导致 token 签发和验证使用不同密钥。  
**影响**: 所有登录请求返回的 token 将无法通过认证中间件验证。  
**建议**: 统一使用 `src/config.py` 的 `JWT_SECRET`，`auth.py` 中 `from src.config import JWT_SECRET`。

### 🟢 LOW

#### A-5: 响应格式兼容层复杂度

**位置**: `src/api/response.py` → `backward_compatible` 装饰器 + `_client_wants_v2` 检测  
**现象**: 同一端点根据请求头 `X-API-Format: v2` 或查询参数 `?format=v2` 返回不同格式。  
**影响**: 增加测试矩阵复杂度，前端需要同时处理两种格式。  
**建议**: 确定迁移时间表，逐步统一切换到 v2 格式。

## 四、依赖

### 🔴 HIGH

#### D-1: `.env` 硬编码生产凭据

**位置**: `.env`  
**内容**:
```
FUXI_JWT_SECRET=fuxi-v1.50-jwt-production-key-change-in-prod
FUXI_ADMIN_DEFAULT_PASSWORD=Fuxi@Admin2026!
MIMO_API_KEY=
```

**风险**:  
1. JWT 密钥明文存储在版本控制中 — 任何有仓库访问权的人可伪造 token
2. 默认管理员密码明文存储 — 首次部署若未修改，存在未授权访问风险
3. `MIMO_API_KEY` 为空 — LLM 服务不可用

**建议**:  
1. `.env` 已在 `.gitignore` 中，但文件已存在于仓库历史 — 需要轮换密钥
2. 首次启动时强制要求设置 `FUXI_JWT_SECRET` 和管理员密码
3. 使用 `docker-compose.yml` 的 `${VAR:?error}` 模式强制环境变量

#### D-2: `requirements.txt` 未锁定版本

**位置**: `requirements.txt`  
**现象**: 使用版本范围（如 `fastapi>=0.139.0,<1.0.0`）而非精确版本。  
**风险**:  
1. 不同环境安装的依赖版本可能不同
2. 次版本升级可能引入 breaking changes
3. 安全漏洞扫描无法精确匹配

**建议**: 使用 `pip freeze > requirements.lock` 生成锁定文件，或迁移到 `pyproject.toml` + `poetry.lock`。

## 五、链路

### 🔴 HIGH

#### L-1: 启动链路单点故障 — 八卦注册失败静默吞掉

**位置**: `src/core/startup.py` → `_register_bagua_guas`  
**现象**:
```python
except (ImportError, AttributeError, RuntimeError) as e:
    logging.getLogger("server").warning(f"[Bagua] {register_name} 注册失败（服务继续启动）: {e}")
```

单个卦注册失败时仅 warning，服务继续启动。如果关键卦（如坤卦 KunGua — 文档管理）注册失败，用户可登录但文档操作全部 500。  
**影响**: 部分功能不可用但无明确错误提示，用户和运维难以定位。  
**建议**:  
1. 关键卦（坤、震、巽）注册失败应阻止启动
2. 非关键卦（离、艮、兑）可降级启动
3. `/api/health` 应返回各卦注册状态

### 🟡 MEDIUM

#### L-2: Redis 连接未实现 — 任务队列不可用

**位置**: `src/infra/task_queue.py` → `get_task_queue()`  
**现象**:
```python
async def get_task_queue() -> RedisStreamTaskQueue:
    global _task_queue
    if _task_queue is None:
        raise RuntimeError("任务队列未初始化，请先调用 initialize_task_queue()")
    return _task_queue
```

`initialize_task_queue(redis_client)` 需要 Redis 客户端，但 `server.py` 的 `startup` 事件中未调用初始化。`documents.py` 的 `upload` 端点调用 `get_task_queue()` 会直接抛出 `RuntimeError`。  
**影响**: 文件上传功能在 Redis 未配置时完全不可用。  
**建议**:  
1. 添加 Redis 连接初始化逻辑（在 `startup.py` 中）
2. 添加降级路径：Redis 不可用时使用内存队列
3. 或在上传端点中添加 task_queue 可用性检查

#### L-3: 健康检查未覆盖关键依赖

**位置**: `src/infra/health_check.py`  
**现象**: `HealthChecker` 支持注册检查项，但未在启动时自动注册以下关键检查：
1. ChromaDB 连接状态
2. Embedder Server (`:8081`) 可用性
3. Rerank Proxy (`:8091`) 可用性
4. Redis 连接状态
5. Ollama (`:11434`) 可用性

`/api/health` 仅返回基本统计，不包含组件级健康状态。  
**建议**: 在 `startup.py` 中注册各组件健康检查，`/api/health` 返回结构化组件状态。

### 🟢 LOW

#### L-4: Feature Flag 文件写入无锁保护

**位置**: `src/services/feature_flags.py` → `set_flag`  
**现象**: Flag 文件写入使用 `json.dumps` + `write_text`，无文件锁。并发写入可能导致数据损坏。  
**建议**: 使用 `fcntl.flock`（Linux）或 `tempfile` + `os.replace` 原子写入。
