## 二、性能验证

### 2.1 aiohttp 连接池

| 组件 | 状态 | 说明 |
|------|------|------|
| `src/core/http_client.py` | 🟢 | 全局单例 `_session`，`aiohttp.ClientSession()` 复用，`get_session()` 检查 `_session.closed` |
| `/api/proxy/loader/*` | 🔴 **HIGH** | **每次请求都 `async with aiohttp.ClientSession() as session`，创建新连接池**，未复用全局 session |
| `/api/antenna/search` (Brave) | 🟡 | 使用 `urllib.request.urlopen()` + `asyncio.to_thread()`，虽然不阻塞事件循环，但未使用 aiohttp 连接池 |

**🔴 HIGH — `/api/proxy/loader/files` 和 `/api/proxy/loader/upload` 连接池未复用**

```python
# src/core/routes.py L155-L170 — 问题代码
async with aiohttp.ClientSession() as session:  # ❌ 每次新建 session
    async with session.get(...) as resp:
        return await resp.json()
```

**修复建议**: 使用 `src.core.http_client.get_session()` 复用全局 session：
```python
from src.core.http_client import get_session
session = await get_session()
async with session.get(...) as resp:
    return await resp.json()
```

### 2.2 同步 I/O 残留

| 位置 | 同步调用 | 状态 | 说明 |
|------|---------|------|------|
| `src/api/files_view.py` L60-80 | `open(fpath, "rb") → f.read()` + `os.walk()` | 🔴 **HIGH** | `view_document()` 和 `download_document()` 是同步函数（无 `async`），在同步上下文中遍历整个 uploads 目录计算 SHA256，**大量文件时会严重阻塞** |
| `src/api/files_view.py` L145-155 | `urllib.request.urlopen()` | 🟡 | Brave 搜索已用 `asyncio.to_thread()` 包装，但定义在 `async antenna_search` 内部，可以接受 |
| `src/category_registry.py` L416 | `requests.post()` | 🟡 | 同步 HTTP 调用，但在 `llm_classify()` 函数中，需确认调用方是否已用 `asyncio.to_thread()` 包装 |
| `src/api/wiki.py` | `engine.get_page()` 等 | 🟢 | 标记了 `# FAKE-ASYNC`，内部同步但快速返回，可接受 |
| `src/api/documents.py` L150 | `tmp_path.write_bytes(content)` | 🟡 | 文件写入在 async 函数中直接执行，大文件可能阻塞事件循环 |

**🔴 HIGH — `files_view.py` 的 `view_document()` 和 `download_document()` 存在严重同步 I/O**

这两个函数遍历整个 uploads 目录，对每个文件计算 SHA256 哈希。如果目录中有大量文件，会阻塞事件循环数秒甚至更久。

**修复建议**:
1. 将函数改为 `async def`，文件操作用 `await asyncio.to_thread()` 包装
2. 建立文件 hash → 路径的索引（SQLite 或内存缓存），避免每次遍历目录

### 2.3 N+1 查询

| 位置 | 问题 | 状态 |
|------|------|------|
| `dashboard.py` | `load_chunks()` 全量加载后在 Python 中聚合统计 | 🟡 可接受（单次查询，非循环） |
| `documents.py` | `load_chunks()` 全量加载后去重 | 🟡 同上 |

```python
"chunk_count": sum(1 for cc in chunks if cc.get("file_hash") == fhash),  # ❌ 对每个文件遍历全部 chunks
```

`_auto_discovery.py` 的 `_SKIP_FILES` 中没有 `system_routes.py`，所以它会被自动发现。同时 `routes.py` 的 `_register_service_routes()` 中没有手动注册 `system_routes.py`（它是通过 `_register_auto_discovered_routes` 注册的），所以实际上是单路径，**无重复**。

但 `files_view.py` 在 `_PREFIX_OVERRIDES` 中映射为 `prefix=""`，意味着它的路由路径（如 `/api/view/{file_hash}`）会被原样注册，这是正确的。

`middleware.py` 中定义了 `setup_middleware(app)` 函数，包含所有中间件配置。但 `server.py` 中也直接注册了相同的中间件。如果 `setup_middleware()` 被调用，会导致**双重注册**。

当前状态：`server.py` 的 `startup()` 事件调用 `register_all_routes(app)` 但**未调用** `setup_middleware(app)`，所以目前只有 `server.py` 中的中间件生效。`middleware.py` 是死代码。

**修复建议**: 
1. 确认 `middleware.py` 是 `server.py` 的替代品还是补充
2. 如果是替代品：将 `server.py` 中的中间件代码迁移到 `middleware.py`，在 `startup()` 中调用 `setup_middleware(app)`
3. 如果是死代码：删除 `middleware.py` 或标记为废弃

| 维度 | 评分 | 说明 |
