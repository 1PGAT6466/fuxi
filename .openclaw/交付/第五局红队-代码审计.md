## 一、代码质量审计

### 1.1 异常吞没（Broad Except / pass）

**严重级别：🔴 HIGH**

代码库中存在 **39 处** `except ...: pass` 模式，异常被完全吞没，无日志、无指标、无恢复。

| 位置 | 异常类型 | 影响 |
|------|---------|------|
| `src/api/synthesis.py:128,136,165` | `except (ImportError, Exception): pass` | **最严重** — `Exception` 兜底 + pass，任何运行时错误都被静默吞没 |
| `src/bagua/xun.py:519` | `except ValueError: pass` | 巽卦逻辑错误被隐藏 |
| `src/bagua/xun.py:702` | `except RuntimeError: pass` | 运行时错误被隐藏 |
| `src/bagua/shutdown.py:250,346,366,386` | `CancelledError/RuntimeError: pass` | 关机流程异常被吞没，可能导致资源泄露 |
| `src/services/llm.py:179` | `except (json.JSONDecodeError, KeyError, IndexError): pass` | LLM 流式响应解析错误被吞没，用户收到空响应无提示 |
| `src/evolution/dream_cycle.py:686` | `except ImportError: pass` | 梦境周期功能静默降级 |
| `src/api/notifications.py:36` | `except (json.JSONDecodeError, IOError): pass` | 通知文件读取失败被吞没 |

**修复建议**：所有 `except: pass` 至少改为 `except XxxError: logger.debug(...)` ，生产关键路径需要 `logger.warning` + 指标计数。

### 1.4 FAKE-ASYNC 函数泛滥

**严重级别：🔴 HIGH**

代码库中存在 **120+ 处** `# FAKE-ASYNC` 标注的函数 — 声明为 `async def` 但内部完全同步执行，无任何 `await`。

**影响**：
- 同步 I/O（SQLite 查询、文件读写）在 async 函数中直接执行，**阻塞事件循环**
- FastAPI 的 async endpoint 如果内部做同步 I/O，会阻塞整个 uvicorn worker
- 高并发时导致请求排队、响应延迟飙升

**高频文件**：

| 文件 | FAKE-ASYNC 数量 |
|------|----------------|
| `src/server.py` | 12 处 |
| `src/api/wiki.py` | 7 处 |
| `src/api/worldtree.py` | 8 处 |
| `src/infra/health_check.py` | 12 处 |
| `src/services/llm.py` | 5 处 |
| `src/taiyang/multi_hop.py` | 7 处 |
| `src/api/admin.py` | 12 处 |

**修复建议**：
1. 对 SQLite 查询使用 `loop.run_in_executor(None, sync_func)` 或 `asyncio.to_thread()`
2. 对文件 I/O 使用 `aiofiles`
3. 将纯同步函数改为普通 `def`，FastAPI 会自动在线程池中运行

### 2.1 路由重复注册

**严重级别：🔴 HIGH**

运行时探测和代码分析发现 **22 个路由路径被重复注册 2-6 次**：

| 路由 | 注册次数 | 注册来源 |
|------|---------|---------|
| `/api/feature-flags/{name}` | **6x** | server.py + routes.py + auto_discovery |
| `/api/metrics` | 3x | server.py + routes.py |
| `/api/eval/run` | 3x | server.py + routes.py |
| `/api/eval/report` | 3x | server.py + routes.py |

| `/` | 3x | server.py + routes.py |
| `/admin` | 3x | server.py + routes.py |
| `/metrics` | 3x | server.py + routes.py |
| `/api/mcp` | 2x | server.py |
| `/api/mcp/call` | 2x | server.py |
| `/api/mcp/tools` | 2x | server.py |
| `/embed` | 2x | routes.py |
| `/health` | 2x | routes.py |
| `/rerank` | 2x | routes.py |

**根因**：`server.py` 手动注册了大量路由，同时 `auto_discover_routers(app)` 和 `core/routes.py` 的 `register_all_routes(app)` 也注册了相同路由。三套注册机制并存。

**风险**：
- 请求可能被错误的 handler 处理（先注册的优先）
- 中间件执行次数异常
- 维护困难，修改一个路由需要改多处

**修复建议**：统一使用 `auto_discover_routers` 或 `register_all_routes` 其中之一，删除冗余注册。

## 三、数据存储审计

### 3.1 SQLite 连接管理 — 连接泄露风险

**严重级别：🔴 HIGH**

**问题 1：散落的 `sqlite3.connect` 调用**

尽管 `src/core/db.py` 定义了统一的 `connect()` 上下文管理器，但代码库中仍有 **45+ 处**直接调用 `sqlite3.connect()`：

| 文件 | 调用次数 | 是否正确关闭 |
|------|---------|------------|
| `src/api/chat.py` | 5 处 | ✅ 使用 `with` 语句 |
| `src/api/auth_routes.py` | 2 处 | ✅ 使用 `with` 语句 |
| `src/bagua/kun.py` | 3 处 | ⚠️ 手动 `conn.close()`，异常时可能泄露 |
| `src/shaoyang/distiller.py` | 3 处 | ⚠️ 无 `with` 语句 |
| `src/shaoyang/relation_builder.py` | 3 处 | ⚠️ 无 `with` 语句 |
| `src/taiyang/graph_router.py` | 1 处 | ⚠️ 无 `with` 语句 |
| `src/taiyang/graph_traversal.py` | 1 处 | ⚠️ 无 `with` 语句 |
| `src/services/retrieval.py:199` | 1 处 | ⚠️ 无 `with` 语句 |
| `src/taiyin/wiki.py` | **12 处** | ⚠️ 大量手动管理，异常路径可能泄露 |
| `src/infra/entity_frontier.py` | 2 处 | ⚠️ `check_same_thread=False` |

**问题 2：`MemoryStore` 长生命周期连接**

`MemoryStore.__init__` 中创建的 `self._db_conn` 是一个 **永不断开的连接**，使用 `check_same_thread=False`。这意味着：
- 多个线程可以同时操作同一连接
- SQLite 的 WAL 模式下读写并发有限
- 连接异常（如 `database is locked`）后无法自动恢复

**修复建议**：
1. 统一使用 `src/core/db.py` 的 `connect()` 上下文管理器
2. `MemoryStore` 改为每次操作获取新连接，或使用 `QueuePool` 连接池
3. `check_same_thread=False` 必须配合线程锁或连接池
