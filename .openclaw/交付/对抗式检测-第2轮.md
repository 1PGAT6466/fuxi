## 1. 深层功能对抗

### 1.1 RAG 检索失败 — ⚠️ 部分成功

**攻击向量**: 构造超长查询 + 特殊字符组合，导致向量嵌入失败或 ChromaDB 查询异常

**代码位置**: `src/services/retrieval.py` → `vector_recall()`, `src/db/vector_store.py` → `query()`

**发现**:
- `ChatRequest` 的 `query` 字段限制 4000 字符，但 `hybrid_search` 内部没有额外长度检查
- `embed_texts()` 调用外部 embedder 服务，超长文本可能导致嵌入服务超时
- ChromaDB `where` 子句接受用户输入的 `category` 参数，未做类型验证

**攻击示例**:
```
POST /api/chat
{"query": "A" * 4000, "history": []}  // 4000字符纯文本
```

**结果**: 部分成功 — 超长查询会被 Pydantic 截断到 4000 字符，但 4000 字符仍可能导致嵌入服务超时。ChromaDB 查询本身有自愈机制（连续失败 3 次触发重连），不会永久损坏。

**修复建议**:
1. 在 `hybrid_search` 入口增加查询长度硬限制（建议 2000 字符）
2. 对 `category` 参数做白名单验证
3. 嵌入服务调用增加 `max_text_length` 截断

### 1.2 LLM 调用超时 — ✅ 成功

**攻击向量**: 构造触发 LLM 长思考的请求，消耗 TokenBudget 并导致超时

**代码位置**: `src/services/llm.py` → `_call_api()`, `call_llm()`

**发现**:
- `_call_api` 有 3 次重试机制，每次重试 `max_tokens` 翻倍（4096 → 8192 → 12288）
- 每次重试都消耗 API 额度，但 TokenBudget 只在调用成功后才计算
- 如果 MiMo 和 DeepSeek 都超时，总等待时间 = `MIMO_TIMEOUT * 3 + DEEPSEEK_TIMEOUT * 1` = 最长 240 秒
- 没有全局超时保护，单个请求可以阻塞事件循环

**攻击示例**:
```
POST /api/chat
{"query": "请详细分析以下代码的每一个潜在安全漏洞，并给出完整的修复方案...(极长prompt)", "history": []}
```

**结果**: 成功 — 可以构造导致 LLM 调用链总耗时超过 4 分钟的请求。虽然每个单独调用有超时，但串行重试机制导致总时间不可控。

**修复建议**:
1. 增加全局请求超时（建议 60 秒），使用 `asyncio.wait_for` 包裹整个 LLM 调用链
2. 重试时不要翻倍 `max_tokens`，保持固定值
3. TokenBudget 应在调用前预检查，而非调用后计算

### 1.3 知识图谱损坏 — ❌ 失败

**攻击向量**: 通过 API 构造导致知识图谱 JSON 文件损坏的操作

**代码位置**: `src/taiyang/graph_traversal.py` → `load_graph()`, `build_adjacency()`

**发现**:
- 知识图谱以 JSON 文件存储（`data/knowledge_graph.json`），只读不写
- `build_adjacency()` 写入 SQLite 邻接表，使用 `INSERT OR IGNORE` 防止重复
- 图遍历函数（`multi_hop_traverse`, `find_paths`）都是只读操作
- 没有公开 API 允许直接修改知识图谱

**结果**: 失败 — 知识图谱是只读的，没有公开写入接口。SQLite 邻接表使用 WAL 模式，有事务保护。

**修复建议**: 无需修复。建议增加图谱文件的完整性校验（如 SHA256 签名）。

### 1.4 ChromaDB 数据不一致 — ⚠️ 部分成功

**攻击向量**: 并发写入导致 ChromaDB 元数据不一致

**代码位置**: `src/db/vector_store.py` → `add()`, `upsert()`, `delete_by_file()`

**发现**:
- ChromaDB 单例模式（`_vector_store` 全局变量），所有请求共享同一个连接
- `add()` 和 `delete_by_file()` 没有事务保护，并发调用可能导致部分写入
- `_clean_metadata()` 截断超长字符串到 1000 字符，但没有去重或验证唯一性
- 向量 ID 由调用方生成，如果调用方传入重复 ID，ChromaDB 会静默覆盖

**攻击示例**:
```
并发上传同一文件两次 → 可能产生重复向量或元数据不一致
```

**结果**: 部分成功 — ChromaDB 内部有锁机制，不会导致数据库损坏。但并发写入可能导致元数据短暂不一致（如文件已删除但向量仍存在）。

**修复建议**:
1. `delete_by_file()` 应该是原子操作（先删除向量，再删除 chunks）
2. 增加写入锁（`asyncio.Lock`）保护并发写入
3. 定期运行一致性校验脚本

## 2. 深层性能对抗

### 2.1 N+1 查询恶化 — ✅ 成功

**攻击向量**: 构造触发大量数据库查询的请求

**代码位置**: `src/api/chat.py` → `chat_session_messages()`, `src/services/retrieval.py` → `hybrid_search()`

**发现**:
- `chat_session_messages()` 返回会话的所有消息，没有分页限制
- `hybrid_search()` 内部创建 4 个并发任务（BM25、向量、Wiki、Table），每个都可能触发数据库查询
- `_l1_2_bm25_search()` 内部调用 `llm_rewrite_query()`，这是一个 LLM 调用，会阻塞
- `_l175_wiki_recall()` 直接执行 SQLite 查询，没有使用连接池

**攻击示例**:
```
GET /api/chat/sessions/{session_id}/messages
// 如果会话有 1000 条消息，每次都返回全部
```

**结果**: 成功 — 可以构造导致大量数据库查询的请求。特别是 `hybrid_search` 的并发任务模型，如果 BM25 和向量检索都触发大量查询，会显著增加延迟。

**修复建议**:
1. `chat_session_messages()` 增加分页参数（`page`, `page_size`）
2. `hybrid_search` 内部的 SQLite 查询应使用连接池
3. 增加查询结果数量上限（如 BM25 最多返回 100 条）

### 2.2 缓存穿透 — ✅ 成功

**攻击向量**: 构造大量不重复的查询，绕过语义缓存

**代码位置**: `src/services/cache.py` → `get_cache()`, `set_cache()`

**发现**:
- L1 缓存使用 MD5 哈希作为 key，精确匹配
- L2 语义缓存使用余弦相似度（阈值 0.92），但每次查询都需要调用 `embed_texts()` 计算嵌入
- 如果查询每次都不同（如包含随机字符串），L1 和 L2 都不会命中
- 每次缓存未命中都会触发完整的 `hybrid_search` 流程

**攻击示例**:
```
POST /api/search
{"q": "随机查询1", "top_k": 15}
POST /api/search
{"q": "随机查询2", "top_k": 15}
... (1000 次不同查询)
```

**结果**: 成功 — 可以构造导致缓存完全失效的请求序列。每次查询都会触发完整的检索流程，包括 LLM 调用、向量检索、BM25 检索等。

**修复建议**:
1. 增加全局请求速率限制（已有，但可以更严格）
2. 对相同用户的短时间大量查询进行检测和限制
3. 考虑使用 Bloom Filter 快速判断查询是否可能命中缓存

### 2.3 连接池耗尽 — ⚠️ 部分成功

**攻击向量**: 并发请求耗尽 SQLite 连接池

**代码位置**: `src/infra/connection_pool.py` → `SQLiteConnectionPool`

**发现**:
- 连接池默认最大 20 个连接
- `get_connection()` 使用上下文管理器，但等待连接释放时使用 `time.sleep(0.1)` 循环 10 次（最多 1 秒）
- 如果所有连接都被占用，新请求会等待 1 秒后抛出异常
- `hybrid_search` 内部的并发任务可能同时请求连接

**攻击示例**:
```
并发发送 25 个搜索请求 → 其中 5 个会等待或失败
```

**结果**: 部分成功 — 连接池有上限保护，不会导致系统崩溃。但并发请求可能导致部分请求超时或失败。

**修复建议**:
1. 使用 `asyncio.Queue` 替代 `time.sleep` 循环，实现异步等待
2. 增加连接池监控指标（活跃连接数、等待队列长度）
3. 考虑使用连接池代理（如 `aiosqlite`）实现真正的异步连接管理

### 2.4 事件循环阻塞 — ⚠️ 部分成功

**攻击向量**: 构造触发同步阻塞操作的请求

**代码位置**: `src/services/retrieval.py`, `src/db/vector_store.py`

**发现**:
- `vector_recall()` 使用 `loop.run_in_executor(None, ...)` 执行 ChromaDB 查询，这是正确的

- `load_chunks()` 使用 `asyncio.to_thread()`，这是正确的

**攻击示例**:

```
触发图谱邻接表重建（需要管理员权限）
```

**结果**: 部分成功 — 大部分 I/O 操作已经正确使用异步方式。但某些同步函数（如 `build_adjacency`）如果在异步上下文中调用，会阻塞事件循环。

**修复建议**:
1. 将 `build_adjacency()` 改为异步函数，或使用 `asyncio.to_thread()` 包裹
2. 审计所有同步函数，确保在异步上下文中使用 `asyncio.to_thread()`

## 3. 深层安全对抗

### 3.1 JWT Token 伪造 — ✅ 成功

**攻击向量**: 尝试伪造 JWT Token，修改 `role` 或 `tenant_id` 字段

**代码位置**: `src/api/auth.py` → `create_jwt_token()`, `verify_jwt_token()`

**发现**:
- JWT 使用 HS256 算法，密钥来自 `FUXI_JWT_SECRET` 环境变量
- `.env` 文件中硬编码了默认密钥: `fuxi-v1.50-jwt-production-key-change-in-prod`
- 如果生产环境没有修改此密钥，攻击者可以使用此密钥签发任意 Token
- Token 包含 `role`、`roles`、`tenant_id` 等敏感字段

**攻击示例**:
```python
import jwt
payload = {
    "sub": "admin",
    "role": "admin",
    "roles": ["admin"],
    "tenant_id": "default",
    "exp": 9999999999
}
token = jwt.encode(payload, "fuxi-v1.50-jwt-production-key-change-in-prod", algorithm="HS256")
```

**结果**: 成功 — 如果生产环境使用默认密钥，攻击者可以伪造任意用户的 Token，包括管理员权限。

**修复建议**:
1. **立即修复**: 确保生产环境设置了强随机密钥（至少 32 字节）
2. 启动时检查密钥强度，拒绝使用弱密钥
3. 考虑使用 RS256 算法（非对称密钥），即使密钥泄露也无法伪造
4. 增加 Token 指纹校验（如绑定 IP 或 User-Agent）

### 3.2 Token 版本绕过 — ❌ 失败

**攻击向量**: 尝试使用旧版本 Token 绕过登出机制

**代码位置**: `src/api/auth.py` → `verify_jwt_token()`, `increment_token_version()`

**发现**:
- Token 包含 `tv`（token version）字段
- `verify_jwt_token()` 检查 `token_tv < current_tv`，如果版本号过期则拒绝
- `increment_token_version()` 在登出/刷新时递增版本号
- 版本号存储在内存中（`token_versions` 字典），重启后重置

**攻击示例**:
```
1. 用户登录，获得 Token (tv=1)
2. 用户登出，tv 递增到 2
3. 攻击者使用旧 Token (tv=1) 访问 → 被拒绝
```

**结果**: 失败 — Token 版本机制工作正常。但注意：服务重启后版本号丢失，旧 Token 会重新生效。

**修复建议**:
1. 将 token_version 持久化到数据库（如 Redis 或 SQLite）
2. 重启后从持久化存储恢复版本号
3. 考虑使用短 Token 有效期（如 15 分钟）+ 刷新机制

### 3.3 多租户数据泄露 — ✅ 成功

**攻击向量**: 通过修改 `X-Tenant-Id` 请求头访问其他租户数据

**代码位置**: `src/auth/tenant.py` → `get_tenant_from_request()`

**发现**:
- `get_tenant_from_request()` 的优先级：`request.state.tenant_id` > JWT payload > `X-Tenant-Id` header > 默认
- 如果中间件正确设置了 `request.state.tenant_id`，则 `X-Tenant-Id` 头会被忽略
- 但某些 API 端点（如 `/api/search`、`/api/rag/search`）没有使用 `get_tenant_from_request()` 进行租户隔离
- 搜索结果没有按租户过滤

**攻击示例**:
```
GET /api/search?q=机密文件&top_k=100
X-Tenant-Id: other-tenant  // 尝试访问其他租户数据
```

**结果**: 成功 — 搜索 API 没有租户隔离，所有租户的数据都可以被搜索到。这是一个严重的多租户隔离漏洞。

**修复建议**:
1. **立即修复**: 在所有数据查询 API 中强制使用租户过滤
2. 在 `hybrid_search` 中增加 `tenant_id` 参数，过滤 ChromaDB 和 SQLite 查询
3. 在 ChromaDB 元数据中存储 `tenant_id`，查询时使用 `where` 子句过滤
4. 定期审计：检查是否有 API 遗漏了租户隔离

### 3.4 权限提升 — ⚠️ 部分成功

**攻击向量**: 普通用户尝试访问管理员 API

**代码位置**: `src/auth/rbac.py`, `src/api/admin.py`

**发现**:
- 管理员 API 使用 `Depends(require_role("admin"))` 保护
- `require_role()` 从 `request.state.user` 获取用户名，然后检查角色
- 如果 `request.state.user` 未设置（如白名单路径绕过），会返回 401
- 但 `require_admin()` 函数检查 `request.state.role`，如果中间件未设置 role，默认为 "user"

**攻击示例**:
```
GET /api/admin/users
Authorization: Bearer <普通用户Token>
// 期望: 403 Forbidden
// 实际: 403 Forbidden (正确)
```

**结果**: 部分成功 — 权限检查机制工作正常，普通用户无法访问管理员 API。但存在一个潜在问题：如果中间件异常导致 `request.state.role` 未设置，`require_admin()` 会默认拒绝，这是安全的。

**修复建议**:
1. 确保所有管理员 API 都使用 `Depends(require_role("admin"))` 或 `Depends(require_admin)`
2. 增加审计日志，记录所有权限拒绝事件
3. 考虑增加 IP 白名单，限制管理员 API 的访问来源

## 4. 前端对抗

### 4.1 前端内存泄漏 — ✅ 成功

**攻击向量**: 构造导致前端 JavaScript 内存泄漏的操作

**代码位置**: `frontend/js/api-client.js` → `__apiCache`, `frontend/js/chat.js` → `chatHistory`

**发现**:
- `__apiCache` 是一个 `Map`，缓存 GET 请求结果，TTL 30 秒
- 缓存没有大小限制，如果大量不同 URL 的 GET 请求，缓存会无限增长
- `chatHistory` 是一个数组，存储所有对话历史，没有大小限制
- `_sessions_store` 和 `_messages_store` 在后端也是无限增长

**攻击示例**:
```
// 前端：不断发起不同 URL 的 GET 请求
for (let i = 0; i < 10000; i++) {
  api(`/api/search?q=test${i}`);
}

// 后端：不断创建新会话
for (let i = 0; i < 10000; i++) {
  POST /api/chat/sessions {"title": "session" + i}
}
```

**结果**: 成功 — 可以构造导致前端和后端内存泄漏的操作。前端 `__apiCache` 没有大小限制，后端 `_sessions_store` 也没有清理机制。

**修复建议**:
1. **前端**: 给 `__apiCache` 增加最大条目数限制（如 1000），使用 LRU 淘汰策略
2. **前端**: 给 `chatHistory` 增加最大长度限制（如 1000 条）
3. **后端**: 给 `_sessions_store` 增加清理机制（如删除 30 天前的会话）
4. **后端**: 增加每个用户的会话数量限制（如 100 个）

### 4.2 XSS 输入 — ⚠️ 部分成功

**攻击向量**: 构造包含恶意脚本的输入，尝试 XSS 攻击

**代码位置**: `src/infra/validation.py` → `sanitize_input()`, `frontend/js/chat.js`

**发现**:
- `sanitize_input()` 只替换了 `<script` 和 `</script` 标签，不完整
- 前端使用 `marked.min.js` 渲染 Markdown，但没有使用 DOMPurify 清理
- 虽然 `dompurify.min.js` 存在于项目中，但没有在所有输出点使用
- CSP 策略允许 `'unsafe-inline'` 的 script-src，削弱了 XSS 防护

**攻击示例**:
```markdown
# 搜索查询

# 或使用事件处理器
<img src=x onerror="alert('XSS')">

# 或使用 Markdown 链接
```

**结果**: 部分成功 — `sanitize_input()` 会拦截 `<script>` 标签，但不会拦截其他 XSS 向量（如事件处理器、javascript: 协议）。前端 `marked.min.js` 可能会渲染恶意 Markdown。

**修复建议**:
1. **立即修复**: 在所有用户输入输出点使用 DOMPurify 清理
2. 更新 `sanitize_input()` 函数，使用白名单策略（只允许安全的 HTML 标签和属性）
3. 移除 CSP 中的 `'unsafe-inline'`，使用 nonce 或 hash
4. 在 `marked.min.js` 配置中启用 HTML 转义

### 4.3 CSRF 请求 — ⚠️ 部分成功

**攻击向量**: 构造跨站请求伪造攻击

**代码位置**: `frontend/js/api-client.js` → CSRF Token 机制

**发现**:
- 前端实现了 CSRF Token 机制（`getCSRFToken()`）
- 每个请求都添加 `X-CSRF-Token` 头
- 但 CSRF Token 存储在 `sessionStorage` 中，与 Token 一起管理
- 后端没有验证 CSRF Token（中间件只检查 JWT Token）

**攻击示例**:
```html
<!-- 恶意网站 -->
<form action="https://fuxi-server/api/chat/send" method="POST">
  <input name="query" value="删除所有文档">
  <input name="session_id" value="xxx">
</form>
<script>document.forms[0].submit();</script>
```

**结果**: 部分成功 — 前端有 CSRF Token 机制，但后端没有验证。如果用户已登录，恶意网站可以构造跨站请求。但由于使用 JWT Token（存储在 sessionStorage），恶意网站无法获取 Token，所以实际风险较低。

**修复建议**:
1. **后端**: 在所有状态变更 API（POST/PUT/DELETE）中验证 `X-CSRF-Token` 头
2. CSRF Token 应该与用户会话绑定，存储在服务器端（如 Redis）
3. 考虑使用 `SameSite=Strict` Cookie 属性
4. 对于 API 服务，可以使用 `Origin` 或 `Referer` 头验证

### 高优先级（需要立即修复）

1. **JWT 默认密钥** — 生产环境必须修改 `FUXI_JWT_SECRET`，否则攻击者可以伪造任意 Token
