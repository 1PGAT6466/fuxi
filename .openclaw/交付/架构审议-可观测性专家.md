# 伏羲系统可观测性架构审计报告

**审计人**：可观测性架构专家  
**审计日期**：2026-07-18  
**审计范围**：伏羲 v1.50 运维层（autonomous/）、基础设施监控（infra/）、可观测性三支柱  

---

## 一、执行摘要

伏羲系统在 v1.50 中已建立初步的可观测性基础，但存在**架构层面的结构性缺陷**：监控职责分散在 `infra/`、`taiyin/`、`services/` 三个层级，缺乏统一的可观测性抽象层；日志未结构化且缺少 trace_id；指标采集覆盖不全；链路追踪完全缺失；告警体系仅有框架未落地。

**核心判断**：当前架构可支撑开发调试，但**无法满足工业化生产环境**的可靠性要求。

## 二、现状分析

|------|------|------|------|

| 请求指标 | `src/infra/request_metrics.py` | HTTP 请求统计 | ✅ 基础可用 |
| 审计日志 | `src/infra/audit_log.py` | 操作审计 | ⚠️ 非结构化 |

| 太阴指标 | `src/taiyin/metrics.py` | Prometheus 定义 | ⚠️ 与 services 重叠 |

### 2.2 四大问题根因

#### 问题 1：日志缺少结构化和 trace_id

**现状**：全部使用 `logging.getLogger()` + f-string 格式化，无 JSON 结构化输出。

```python
# 当前写法（遍布全系统）
logger.error(f"[Monitor] 未处理异常: {exc}")
logger.info(f"[Monitor] 错误处理器已设置")
```

**影响**：
- 无法按 trace_id 串联一次请求的完整链路
- 日志检索依赖 grep，效率极低
- 无法接入 ELK/Loki 等日志系统

#### 问题 2：指标采集不完整

**现状**：
- `MetricsCollector` 是纯内存实现，进程重启即丢失
- `prometheus_metrics.py` 定义了指标但未在所有关键路径埋点
- RAG 核心指标（检索召回率、rerank 命中率、LLM 首 token 延迟）缺失
- 多进程/多机部署下指标无法聚合

**影响**：
- 无法建立 SLI/SLO 体系
- 性能瓶颈定位困难

#### 问题 3：链路追踪缺失

**现状**：系统中无 OpenTelemetry、Jaeger、Zipkin 或任何分布式追踪集成。

**影响**：
- 一次 RAG 请求经过 shaoyang→taiyang→shaoyin→taiyin 四个系统，无法追踪各阶段耗时
- 跨机调用（本机→服务器→装载机）完全黑盒
- 性能问题无法精确定位到具体器官

#### 问题 4：告警规则不完善

**现状**：`prometheus_metrics.py` 中提到了告警规则但仅为框架代码，无实际告警通道和规则配置。

**影响**：
- 服务异常只能被动发现
- 无 on-call 机制

## 三、架构审议

### 3.1 当前运维架构是否合理？

**结论：不合理，存在三个结构性问题。**

#### 问题 A：职责分散，边界模糊

当前监控相关代码分布在三个层级：

```
infra/          ← 基础设施监控（系统指标、审计、健康检查）
services/       ← 业务指标（Prometheus 定义）
taiyin/         ← 对外监控（错误处理、审计、指标暴露）
```

**问题**：同一个"审计日志"功能在 `infra/audit_log.py` 和 `taiyin/audit.py` 中各实现了一份。同一个"Prometheus 指标"在 `services/metrics.py`、`services/prometheus_metrics.py`、`taiyin/metrics.py` 中有三处定义。这违反了**单一职责原则**，导致维护成本高、一致性差。

#### 问题 B：缺少可观测性抽象层

当前每个模块直接调用 `logging`、`prometheus_client`、`psutil`，没有统一的可观测性门面（Facade）。如果要切换日志框架或指标后端，需要修改所有模块。

#### 问题 C：运维自动化模块（autonomous/）完全缺失

架构文档中提到了 autonomous/ 层（scheduler、monitor、healer、sync、reporter），但代码库中**实际不存在这些模块**。这意味着：
- 无定时调度能力（scheduler）
- 无自动健康巡检（monitor）
- 无自动故障恢复（healer）
- 无数据同步监控（sync）
- 无自动报告生成（reporter）

### 3.2 如何设计运维模块边界？

**原则：每个模块职责单一，输入输出明确，不跨层调用。**

#### 推荐的四层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    展示层 (Presentation)                      │
│  reporter — 生成运维报告、仪表盘数据、趋势分析                  │
└─────────────────────────────────────────────────────────────┘
                              ▲
┌─────────────────────────────────────────────────────────────┐
│                    决策层 (Decision)                          │
│  healer — 根据告警和策略执行自动修复                            │
│  alerter — 告警规则评估、通知分发                               │
└─────────────────────────────────────────────────────────────┘
                              ▲
┌─────────────────────────────────────────────────────────────┐
│                    采集层 (Collection)                        │
│  monitor — 健康检查、指标采集、日志收集                          │
│  tracer — 分布式链路追踪                                      │
│  scheduler — 定时任务调度                                     │
└─────────────────────────────────────────────────────────────┘
                              ▲
┌─────────────────────────────────────────────────────────────┐
│                    基础层 (Foundation)                        │
│  logger — 结构化日志（JSON + trace_id）                        │
│  metrics — 指标注册与暴露（Prometheus）                         │
│  event_bus — 事件总线（经络系统）                               │
└─────────────────────────────────────────────────────────────┘
```

#### 模块边界定义

| 模块 | 输入 | 输出 | 不做什么 |
|------|------|------|----------|
| **logger** | 日志事件 | 结构化 JSON 日志流 | 不做日志存储和检索 |
| **metrics** | 指标数据 | Prometheus 格式 | 不做告警判断 |
| **tracer** | 请求上下文 | 链路 span | 不做性能分析 |
| **monitor** | 系统/服务状态 | 健康状态 + 指标 | 不做自动修复 |
| **alerter** | 指标阈值/规则 | 告警通知 | 不做指标采集 |
| **healer** | 告警事件 | 修复动作 | 不做告警判断 |
| **scheduler** | 定时任务定义 | 任务执行 | 不做任务逻辑 |
| **sync** | 同步状态 | 同步结果 | 不做数据转换 |
| **reporter** | 历史数据 | 报告/仪表盘 | 不做实时监控 |

## 四、具体改进建议

### 4.1 结构化日志（优先级：P0）

**目标**：所有日志输出 JSON 格式，自动注入 trace_id、service_name、module。

```python
# src/infra/structured_logger.py
import logging
import json
import uuid
from contextvars import ContextVar
from datetime import datetime, timezone

# 请求级 trace_id，通过 ContextVar 跨协程传播
_trace_id: ContextVar[str] = ContextVar("trace_id", default="")

def get_trace_id() -> str:
    return _trace_id.get()

def set_trace_id(trace_id: str = None) -> str:
    tid = trace_id or uuid.uuid4().hex[:16]
    _trace_id.set(tid)
    return tid

class StructuredFormatter(logging.Formatter):
    """JSON 结构化日志格式化器"""

    def format(self, record: logging.LogRecord) -> str:
        log_entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
            "trace_id": get_trace_id(),
            "module": record.module,
            "function": record.funcName,
            "line": record.lineno,
        }

        # 附加异常信息
        if record.exc_info and record.exc_info[1]:
            log_entry["exception"] = {
                "type": type(record.exc_info[1]).__name__,
                "message": str(record.exc_info[1]),
            }

        # 附加额外字段
        if hasattr(record, "extra_data"):
            log_entry["data"] = record.extra_data

        return json.dumps(log_entry, ensure_ascii=False)


def setup_structured_logging(service_name: str = "fuxi"):
    """全局初始化结构化日志"""
    root = logging.getLogger()
    root.setLevel(logging.INFO)

    # 移除默认 handler
    for h in root.handlers[:]:
        root.removeHandler(h)

    # JSON handler
    handler = logging.StreamHandler()
    handler.setFormatter(StructuredFormatter())
    root.addHandler(handler)

    # 文件 handler（按天轮转）
    from logging.handlers import TimedRotatingFileHandler
    file_handler = TimedRotatingFileHandler(
        filename=f"logs/{service_name}.json",
        when="midnight",
        backupCount=30,
        encoding="utf-8",
    )
    file_handler.setFormatter(StructuredFormatter())
    root.addHandler(file_handler)
```

**FastAPI 中间件集成**：

```python
# src/middleware/trace_middleware.py
from starlette.middleware.base import BaseHTTPMiddleware
from src.infra.structured_logger import set_trace_id

class TraceMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        # 从请求头提取或生成 trace_id
        trace_id = request.headers.get("X-Trace-Id") or set_trace_id()
        set_trace_id(trace_id)

        response = await call_next(request)
        response.headers["X-Trace-Id"] = trace_id
        return response
```

### 4.2 链路追踪（优先级：P0）

**目标**：集成 OpenTelemetry，追踪 RAG 请求全链路。

```python
# src/infra/tracing.py
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource

def init_tracing(service_name: str = "fuxi", otlp_endpoint: str = "localhost:4317"):
    """初始化 OpenTelemetry 链路追踪"""
    resource = Resource.create({"service.name": service_name})
    provider = TracerProvider(resource=resource)

    # OTLP 导出器（兼容 Jaeger/Tempo/Zipkin）
    exporter = OTLPSpanExporter(endpoint=otlp_endpoint, insecure=True)
    provider.add_span_processor(BatchSpanProcessor(exporter))

    trace.set_tracer_provider(provider)
    return trace.get_tracer(service_name)


# 在 RAG 关键路径中使用
tracer = init_tracing()

async def rag_search(query: str):
    with tracer.start_as_current_span("rag_search") as span:
        span.set_attribute("query", query)

        with tracer.start_as_current_span("retrieve"):
            results = await retrieve(query)

        with tracer.start_as_current_span("rerank"):
            ranked = await rerank(results)

        with tracer.start_as_current_span("generate"):
            answer = await generate(ranked)

        span.set_attribute("result_count", len(ranked))
        return answer
```

#### 告警分级

| 级别 | 条件 | 响应时间 | 通知渠道 |
|------|------|----------|----------|
| P0-Critical | 服务不可用、数据丢失 | 5 分钟 | 电话 + 企微 |

### 4.5 autonomous/ 模块实现建议（优先级：P2）

#### scheduler — 调度器

```python
# src/autonomous/scheduler/scheduler.py
"""
职责：定时任务调度
输入：任务定义（cron 表达式 + 回调）
输出：任务执行结果
边界：不做任务逻辑，只管调度
"""
import asyncio
from croniter import croniter
from datetime import datetime

class TaskScheduler:
    def __init__(self):
        self._tasks = {}

    def register(self, name: str, cron_expr: str, callback):
        self._tasks[name] = {
            "cron": croniter(cron_expr, datetime.now()),
            "callback": callback,
            "last_run": None,
        }

    async def run_loop(self):
        while True:
            now = datetime.now()
            for name, task in self._tasks.items():
                if task["cron"].get_next(datetime) <= now:
                    await task["callback"]()
                    task["last_run"] = now
            await asyncio.sleep(10)
```

#### healer — 自修复

```python
# src/autonomous/healer/healer.py
"""
职责：根据告警事件执行预定义的修复动作
输入：告警事件（来自经络系统）
输出：修复结果
边界：只执行已注册的修复策略，不自行判断告警
"""
class Healer:
    STRATEGIES = {
        "service_down": ["restart_service"],
        "high_memory": ["clear_cache", "restart_service"],
        "disk_full": ["cleanup_temp", "rotate_logs"],
        "sync_lag": ["force_sync"],
    }

    async def on_alert(self, signal: Signal):
        alert_type = signal.payload.get("type")
        strategies = self.STRATEGIES.get(alert_type, [])

        for strategy in strategies:
            result = await self._execute(strategy)
            if result["success"]:
                break
```
