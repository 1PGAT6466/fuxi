## 一、功能对抗

### 1.1 构造导致前端崩溃的输入

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| F-1 | **超长查询注入** — 发送 100KB+ 的 query 到 `/api/search?q=...` | ⚠️ 部分成功 | 前端 `Search.vue` 直接将 `data.results` 渲染到 DOM，无长度截断。超长 `file_name` 或 `snippet` 可能导致 DOM 渲染卡顿。但后端 `ChatRequest.validate_query` 有 4000 字符限制（仅 chat 端点）。 |
| F-2 | **XSS payload 在搜索结果中** — 上传含 `<script>alert(1)</script>` 的文件名文档 | ⚠️ 部分成功 | 搜索结果的 `file_name` 字段直接来自上传文件名，未做 HTML 转义。前端 Vue3 默认对 `{{ }}` 插值转义，但 `v-html` 或手动 DOM 操作场景下可能执行。**需检查前端是否有 `v-html` 使用。** |
| F-3 | **畸形 JSON 历史消息** — 在 `history` 字段中注入非标准结构 | ✅ 失败 | Pydantic `field_validator` 对 `history` 做了 50 条上限校验，`extra = "ignore"` 忽略多余字段。 |
| F-4 | **空 query + 大量 history** — 边界条件测试 | ✅ 失败 | `validate_query` 要求非空，`validate_history` 限制 50 条。 |

### 1.2 构造导致后端 500 的请求

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| F-5 | **SQLite 并发写入** — 同时发送多个删除/上传请求 | ⚠️ 部分成功 | `load_chunks()` / `save_chunks()` 使用同步 `sqlite3.connect`，无 WAL 模式配置。并发写入可能触发 `database is locked` 错误，导致 500。**建议：启用 WAL 模式 + 连接池。** |
| F-6 | **无效 file_hash 删除** — `/api/documents/../../../etc/passwd` | ✅ 失败 | 路由参数 `{file_hash}` 由 FastAPI 自动 URL decode，不会包含 `/`。`documents.py` 的删除逻辑按精确 hash 匹配，不涉及文件系统路径。 |
| F-7 | **OSError 触发** — 上传只读目录下的同名文件覆盖 | ✅ 失败 | `documents.py:upload()` 有完整的 `try/except (OSError, IOError, ValueError)` 捕获，返回 500 + 错误信息而非崩溃。 |
| F-8 | **delete_document 物理文件删除中的 async 嵌套** | 🔴 成功 | `documents.py:delete_document()` 中 `_delete_physical_files()` 内部使用 `await asyncio.to_thread(lambda: open(fpath, "rb").read())`，但 `_delete_physical_files` 本身是普通函数被 `await asyncio.to_thread()` 调用。**Lambda 内的 `await` 会在同步上下文中引发 RuntimeError。** 这是一个真实 bug。 |

### 1.3 绕过认证访问受保护端点

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| F-9 | **无 Token 访问 /api/admin/stats** | ✅ 失败 | `AuthMiddleware` 对所有非白名单路径强制 Bearer Token 验证。`require_role("admin")` 依赖进一步检查角色。 |
| F-10 | **伪造 JWT Token** — 使用弱密钥猜测 | ✅ 失败 | `config.py` 在未设置 `FUXI_JWT_SECRET` 时自动生成 64 字符随机 hex 密钥。HS256 + 随机密钥安全性可接受。 |
| F-11 | **白名单路径遍历** — `/static/../../etc/passwd` | ⚠️ 部分成功 | `_is_whitelisted()` 仅检查 `path.startswith("/static/")`，不做路径规范化。**但 FastAPI 的静态文件挂载会自行处理路径规范化**，实际风险取决于静态文件服务的实现。 |
| F-12 | **Token 版本号绕过** — 使用旧版本 Token | ✅ 失败 | `verify_jwt_token()` 检查 `token_version`，递增后旧 Token 立即失效。黑名单机制也已实现。 |
| F-13 | **view_document 匿名访问** | 🔴 成功 | `files_view.py:view_document()` 中，当 `request.state.user` 未设置时，回退为 `user = "anonymous"` 并允许访问。**这意味着白名单路径（如 `/static/`）触发的请求可以绕过认证查看文档。** 虽然 `AuthMiddleware` 应拦截 `/api/view/`，但代码中的"安全降级"逻辑是一个隐患。 |

### 1.4 并发请求导致竞态条件

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| F-14 | **并发上传同名文件** | ⚠️ 部分成功 | `upload()` 使用 `os.path.basename` + `resolve()` 做路径穿越防护，但无文件锁。两个并发请求可能同时写入同一临时文件，导致数据损坏。 |
| F-15 | **并发删除同一文档** | ⚠️ 部分成功 | `delete_document()` 先 load_chunks 再 save_chunks，无原子操作保护。两个并发删除可能导致部分 chunks 被恢复（后写入的 save 覆盖前一个的删除）。 |

## 二、性能对抗

### 2.1 构造导致内存泄漏的请求

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| P-1 | **Token 黑名单无限增长** | ✅ 失败 | `_cleanup_blacklist()` 在每次 `_blacklist_token()` 时清理过期条目。但需注意：如果大量 Token 同时到期前涌入，内存仍会暂时膨胀。 |
| P-2 | **限流器 `_limiters` 字典无限增长** | 🔴 成功 | `InputLimitMiddleware._limiters` 使用 `dict` 存储，key 为 `f"global:{ip}"`。**攻击者使用不同 IP（或伪造 IP）发送请求，每个 IP 创建一个 `SlidingWindowRateLimiter`，永不清理。** 长期运行会导致内存缓慢增长。 |
| P-3 | **`_token_blacklist` 内存存储** | ⚠️ 部分成功 | 服务重启后黑名单重置，但运行期间大量登出操作会积累条目。`_cleanup_blacklist()` 仅在 `_blacklist_token()` 调用时触发，无定时清理。 |

### 2.2 构造导致 CPU 飙升的请求

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| P-4 | **大量搜索请求** — 每次触发 BM25 + ChromaDB 检索 | ⚠️ 部分成功 | 全局限流 60/min 可缓解，但搜索本身无单独限流。`hybrid_search` 触发 jieba 分词 + 向量检索 + 可能的 Rerank，单次请求 CPU 开销较大。 |
| P-5 | **知识图谱全量查询** — `/api/graph` 不传 entity 参数 | ⚠️ 部分成功 | 返回整个 `knowledge_graph.json` 的所有 nodes 和 edges。如果图谱很大（数万节点），JSON 序列化 + 网络传输会消耗大量 CPU 和带宽。**建议添加分页。** |
| P-6 | **Prompt 注入检测正则 ReDoS** | ✅ 失败 | `INJECTION_PATTERNS` 使用的正则表达式相对简单，无嵌套量词，不存在 ReDoS 风险。 |

### 2.3 构造导致连接耗尽的请求

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| P-7 | **慢速连接攻击 (Slowloris)** | ⚠️ 部分成功 | uvicorn 配置 `timeout_keep_alive=120`（2分钟），`h11_max_incomplete_event_size=524288000`（500MB）。攻击者可缓慢发送请求头，占用连接槽位长达 2 分钟。**建议降低 keep_alive 超时。** |
| P-8 | **WebSocket 连接耗尽** — `/api/feature-flags/ws` | ⚠️ 部分成功 | WebSocket 端点无连接数限制。攻击者可建立大量 WebSocket 连接耗尽服务器资源。 |

### 2.4 构造导致磁盘满的请求

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| P-9 | **大量小文件上传** — 每次上传刚好低于 200MB 限制 | ⚠️ 部分成功 | `MAX_FILE_MB` 默认 200MB，但不限制总上传次数或总磁盘使用量。持续上传可填满磁盘。**建议添加磁盘配额检查。** |
| P-10 | **日志文件膨胀** | ✅ 失败 | `RotatingFileHandler` 配置 `maxBytes=10MB, backupCount=5`，最多占用 60MB。审计日志 `audit.jsonl` 无 rotation，但增长较慢。 |

## 三、安全对抗

### 3.1 SQL 注入

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| S-1 | **搜索参数注入** — `/api/search?q='; DROP TABLE chunks;--` | ✅ 失败 | `hybrid_search` 使用 ChromaDB 和 jieba BM25，不直接拼接 SQL。SQLite 查询使用参数化查询（需确认 `data_store.py`）。 |
| S-2 | **file_hash 注入** — `/api/documents/' OR '1'='1` | ✅ 失败 | FastAPI 路径参数自动处理 URL 编码，`file_hash` 作为字符串精确匹配，不拼接 SQL。 |
| S-3 | **chat history 注入** — 在 history 中嵌入 SQL 片段 | ✅ 失败 | history 作为 Python dict 传递给 LLM，不接触 SQL 层。 |

### 3.2 XSS 攻击

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| S-4 | **存储型 XSS** — 上传文件名为 `<img onerror=alert(1)>` 的文件 | ⚠️ 部分成功 | 文件名原样存储到 chunks.db，搜索结果返回时未做 HTML 转义。Vue3 默认对 `{{ }}` 转义，但前端如果使用 `v-html` 渲染搜索结果，则存在 XSS 风险。**安全头 `X-XSS-Protection: 1; mode=block` 可部分缓解。** |
| S-5 | **反射型 XSS** — `/api/search?q=<script>alert(1)</script>` | ✅ 失败 | API 返回 JSON，不直接渲染 HTML。前端使用 `fetch` 获取 JSON 数据，非服务端渲染。 |
| S-6 | **CSP 绕过** — 利用 `'unsafe-inline'` 执行脚本 | ⚠️ 部分成功 | CSP 策略 `script-src 'self' 'unsafe-inline'` 允许内联脚本执行。**如果攻击者能注入内联 `<script>` 标签（通过文件名 XSS），CSP 无法阻止执行。** |

### 3.3 CSRF 攻击

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| S-7 | **跨站请求伪造** — 从恶意网站发起 POST /api/chat | ✅ 失败 | API 使用 Bearer Token 认证（非 Cookie），CORS 策略限制 `allow_origins` 为 localhost。恶意网站无法获取 Token，也无法通过 CORS 预检。 |
| S-8 | **CORS 配置过宽** — `allow_origins` 是否包含 `*` | ✅ 失败 | `CORS_ORIGINS` 默认为 `http://localhost:8080,http://127.0.0.1:8080`，未使用通配符。 |

### 3.4 文件上传攻击

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| S-9 | **路径穿越上传** — 文件名为 `../../etc/cron.d/backdoor` | ✅ 失败 | `documents.py:upload()` 使用 `os.path.basename()` 剥离路径 + `resolve()` 规范化 + `startswith()` 校验。三重防护有效。 |
| S-10 | **可执行文件上传** — 上传 `.py`, `.sh`, `.bat`, `.ps1` 文件 | 🔴 成功 | `ALLOWED_EXTENSIONS` 白名单包含 `.py`, `.js`, `.sh`, `.bat`, `.ps1` 等可执行扩展名。**攻击者可上传恶意脚本文件。** 虽然文件不会被自动执行，但如果其他路径（如 `/api/view/`）允许下载并由用户执行，存在社会工程攻击风险。 |
| S-11 | **双扩展名绕过** — `malware.txt.exe` | ✅ 失败 | 白名单检查的是最后一个扩展名（`os.path.splitext`），`.exe` 不在白名单中。 |
| S-12 | **超大文件 DoS** — 上传 200MB 文件消耗内存 | ⚠️ 部分成功 | `file.read()` 将整个文件读入内存。并发上传多个 200MB 文件可耗尽服务器内存。**建议使用流式写入。** |

### 3.5 路径遍历攻击

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| S-13 | **view_document 路径遍历** — `/api/view/../../etc/passwd` | ✅ 失败 | `view_document()` 使用 `os.walk()` 遍历上传目录，按 hash 匹配文件内容，不直接拼接用户输入到路径。 |
| S-14 | **静态文件路径遍历** — `/static/../../etc/passwd` | ⚠️ 部分成功 | 取决于 FastAPI 的 `StaticFiles` 配置是否启用 `html=True` 和路径规范化。默认 `StaticFiles` 会阻止 `..` 遍历，但建议显式测试。 |

## 四、架构对抗

### 4.1 绕过中间件

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| A-1 | **OPTIONS 请求绕过认证** | ✅ 失败（设计如此） | `AuthMiddleware` 对 `OPTIONS` 请求直接放行，这是 CORS preflight 的标准行为。攻击者无法通过 OPTIONS 获取数据。 |
| A-2 | **中间件顺序攻击** | ⚠️ 部分成功 | `setup_middleware()` 的注册顺序为：安全头 → AuthMiddleware → InputLimitMiddleware → CORS → GZip → 引擎路由 → 指标。**AuthMiddleware 在 CORS 之前注册，意味着认证失败的请求不会返回 CORS 头，可能导致前端调试困难。** 但这不是安全漏洞。 |
| A-3 | **异常处理器信息泄露** | ✅ 失败 | 生产环境下，`global_http_exception_handler` 对 500+ 错误返回通用消息"请求处理失败"，不暴露内部细节。开发环境才返回 `str(exc.detail)`。 |

### 4.2 绕过限流

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| A-4 | **伪造 IP 绕过限流** — `X-Forwarded-For` 头 | 🔴 成功 | `InputLimitMiddleware` 使用 `request.client.host` 获取 IP，但 **未配置信任的代理层**。如果部署了反向代理（nginx），`request.client.host` 始终为代理 IP（127.0.0.1），限流形同虚设。如果未部署代理，攻击者可通过 `X-Forwarded-For` 头伪造不同 IP（取决于 uvicorn 的 `proxy_headers` 配置）。 |
| A-5 | **限流器内存增长** — 不同 IP 创建无限限流器 | 🔴 成功 | 同 P-2。`_limiters` 字典按 IP 分 key，无过期清理机制。 |
| A-6 | **slowapi + InputLimitMiddleware 双重限流冲突** | ⚠️ 部分成功 | `middleware.py` 同时注册了 `slowapi` 限流（60/min）和 `InputLimitMiddleware`（60/min）。两套限流独立运行，可能导致不一致行为。**建议统一为一套限流机制。** |

### 4.3 绕过 CORS

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| A-7 | **CORS 配置环境变量注入** — 通过 `.env` 修改 `KB_CORS_ORIGINS` | ⚠️ 部分成功 | 如果攻击者能修改 `.env` 文件（需服务器权限），可将 `CORS_ORIGINS` 设为 `*`。但这需要更高的攻击权限。 |
| A-8 | **WebSocket 绕过 CORS** | ⚠️ 部分成功 | WebSocket 连接不受 CORS 限制。`/api/feature-flags/ws` 端点可通过 WebSocket 从任意来源连接。**但 WebSocket 仍需认证 Token。** |

### 4.4 绕过 CSP

| # | 攻击向量 | 结果 | 详情 |
|---|---------|------|------|
| A-9 | **CSP `'unsafe-inline'` 绕过** | 🔴 成功 | CSP `script-src 'self' 'unsafe-inline'` 允许所有内联脚本。**这是 CSP 的重大削弱。** 如果攻击者能注入 HTML（如通过存储型 XSS），可直接执行任意 JavaScript。**建议使用 nonce 或 hash 替代 `'unsafe-inline'`。** |
| A-10 | **CSP `connect-src` 过宽** — `https:` 允许所有 HTTPS 连接 | ⚠️ 部分成功 | `connect-src 'self' http://localhost:* ws://localhost:* https:` 中的 `https:` 允许向任意 HTTPS 地址发请求。攻击者注入的脚本可将窃取的数据外发到攻击者服务器。**建议限制为已知域名。** |

## 五、发现汇总

### 严重级别（🔴 成功）

| F-8 | 功能 | `delete_document` 物理文件删除中 async 嵌套 bug | 删除文档时 500 错误 | 将 `_delete_physical_files` 改为纯同步函数，不使用 `asyncio.to_thread` 包裹内部的 `await` |
