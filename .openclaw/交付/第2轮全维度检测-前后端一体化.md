## 2. 性能

### 2.1 后端性能

#### 2.1.1 异步函数使用率

**检测结果**: 
- API 路由文件: 32 个
- 使用 `async def` 的路由: **100%** ✅
- 标记为 `FAKE-ASYNC` 的函数: 约 40%（内部同步执行，仅为接口统一）

**问题**:
- 🟡 **MEDIUM**: 大量函数标记为 `FAKE-ASYNC`，内部使用同步代码
  - 影响：阻塞事件循环，降低并发性能
  - 示例：`admin.py` 的 `admin_stats()`, `dashboard.py` 的 `dashboard()`
  - 建议：将同步 I/O 操作改为 `await asyncio.to_thread()`

#### 2.1.2 缓存命中率

**检测结果**:
- **L1 缓存** (精确匹配): `cache.py` 实现，MAX_CACHE_SIZE=200
- **L2 缓存** (语义相似): 阈值 0.92，自动过期
- **Chunk 缓存**: `data_store.py` 实现，动态 TTL (30-120秒)

**问题**:
- 🟢 **LOW**: 缓存实现完善，有 LRU 淘汰和过期机制
- 🟡 **MEDIUM**: 缓存命中率指标未暴露到 `/api/cache/stats` 端点（需检查实现）

#### 2.1.3 连接池使用

**检测结果**:
- **SQLite 连接池**: `connection_pool.py` 实现
  - 最大连接数: 5
  - WAL 模式: ✅
  - 线程安全: ✅
- **ChromaDB 连接**: 单例模式，全局复用
- **HTTP 连接**: `aiohttp.ClientSession` 每次请求新建（❌ 性能问题）

**问题**:
- 🔴 **HIGH**: `llm.py` 的 `_call_api()` 每次调用都创建新的 `aiohttp.ClientSession`
  - 影响：大量并发请求时连接数暴增，可能导致连接耗尽
  - 建议：使用全局连接池或复用 Session

#### 2.1.4 异步任务

**检测结果**:
- **任务队列**: `task_queue.py` 实现
  - 支持: 文件处理、评测、学习
  - 异步发布: ✅
- **后台任务**: Dream Cycle、评测自动化

**问题**:
- 🟢 **LOW**: 异步任务实现完善
